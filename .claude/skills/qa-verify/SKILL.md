# QA Verify — 測試與驗收守衛

規則細節見 `docs/TESTING.md`、`docs/AI_RULES.md` 第 10 節（完成的定義）。
需要完整審查視角時，加讀角色完整版 `../../../docs/roles/qa-engineer.md`。

## 鐵律

1. **金流、權限、交易三類必須有自動化測試，不可豁免**（TESTING 第 2 節）。
2. 錯誤流程與 happy path 一樣要測：邊界值、重複提交、越權存取、無效狀態。
3. 修一個 bug，必附一個能重現它的測試（或手動重現步驟），修復後轉綠才算修好。
4. 改到關鍵路徑時跑回歸清單：登入 / 核心流程 / 結帳一筆 / webhook / 後台讀寫。
5. 不得用 production 資料測試；測試資料要可識別、測後清理。
6. 「實際結果」欄位只有 AI 真的跑過流程才能填。

## 獨立審查（中改以上集中審查 / 高風險任務交付前）

完整規則與回報格式見 `docs/ROLE_SYSTEM.md` §7（單一事實來源，不重抄）。鐵律摘要：

1. **執行載體是乾淨的獨立 AI 實例／子代理**：只給 diff＋需求、不給實作脈絡
   （「同一 AI 分飾多角開會」已退役——另一顆腦，不是另一頂帽子）。
   機器化入口 `bash scripts/review.sh`（高風險範圍；目前軟跑，不掛硬門）。
2. **實作者迴避**：實作者不得批准自己的成果，只說明與補證據。
3. 審查按視角（資安 / 資料庫 / 金流 / QA / UX）分開回報，禁止「同上」式帶過。
4. **否決權**：資安 / 資料庫 / 金流 / 核心需求 / 嚴重手機 UX 的阻擋，不得被說服蓋過。
5. 無證據不得稱通過，一律標 `Unverified`；也禁止虛構風險湊數。

## 回報鐵律（三個詞不得混用）

| 用語 | 條件 |
|---|---|
| 「完成」 | build / 測試 / preflight / 實跑流程 / 文件同步全過 |
| 「已部署，待驗證」 | build/push 過了，真實流程還沒跑 |
| 「已修改，待驗證」 | 改了但連部署驗證都還沒有 |

使用者待辦的前置步驟（待套 SQL、待實測付款）必列回報最上方。

## 交付格式（手動驗證）

```text
驗證項目：
前置條件：
操作步驟：1. … 2. … 3. …
預期結果：
實際結果：（AI 實際跑過才能填）
```
