# 《NLP人性溝通術計分系統》遊戲機制與系統效能優化建議報告

本報告針對系統當前的**遊戲化經驗值（EXP）設計**與**軟體系統效能（Supabase / React）**進行深度分析，列出可優化的方向、背後原因及具體實作方案，協助團隊打造兼具「高度黏著感」與「極致流暢度」的系統。

---

## 🎯 第一部分：遊戲化經驗值（EXP）設計優化與原因

在遊戲化設計（Gamification）中，最核心的目標是讓學員產生 **「即時回饋感（Feedback Loop）」** 與 **「目標趨近效應（Goal Gradient Effect）」**。以下是針對 EXP 與任務點數的優化建議與原因：

### 1. 調整升級門檻為扁平 700 EXP / 級（已實作）
*   **調整原因**：
    *   **防止後期膨脹與跳級過快**：原先每級 500 EXP 門檻較低。後期學員因為累積了許多高分任務（例如：推薦初階 +1500 EXP、次感元個案 +1000 EXP），若維持 500 EXP，一旦審核通過，學員會瞬間連升 3 到 5 級，導致神獸「最高階形態（Level 25）」太容易達到，降低了稀有度與後期追求的動力。
    *   **尊榮感與終極目標**：設定為 700 EXP 後，終極形態需要累計達 $25 \times 700 = 17,500$ 經驗值，使其成為期末真正「卓越修行者」的榮譽象徵，拉長遊戲生命週期。

### 2. 雙軌天數加速預估與「時間折抵」機制（已實作）
*   **調整原因**：
    *   **消除前期挫折感**：若單純以每日保底打卡速度估算，破殼孵化（3500 EXP）需要 24 天，這會讓新學員覺得目標遙不可及，在第一週便失去打卡動力。
    *   **激勵性雙軌顯示**：改為「積極挑戰最快 5 天破殼，認真修行也僅需 12 天」，利用時間框架縮短心理阻力。
    *   **蔡加尼克效應（Zeigarnik Effect）與時間減免**：在攻略中條列「每週實作任務 ➔ 時間立減 1.5 天」、「推薦初階 ➔ 時間立減 4.2 天」。學員為了想「提早少去幾天」看到神獸孵化，會主動且迫切地去挑戰高難度任務，從「被動填寫」轉為主動推廣。

### 3. 建議進階優化：漸進式經驗值曲線 (Progressive EXP Curve)
若要讓前期的升級感更爽快，後期更有挑戰性，建議未來可將扁平的 700 EXP 升級公式，調整為「前快後慢」階梯式：
*   **LV.1 ~ LV.5 (破殼期)**：每級僅需 **300 EXP**（累計 1,500 EXP 即可孵化幼獸，**最快 2-3 天**達成，前期回饋極快）。
*   **LV.6 ~ LV.15 (成長期)**：每級需要 **600 EXP**。
*   **LV.16 ~ LV.25 (終極修煉期)**：每級需要 **1,000 EXP**。
*   *好處*：既保證了前期的快速破殼爽快感，又鎖定了後期終極形態的超高含金量。

### 4. 建議進階優化：連續修行加成機制 (Streak Multiplier)
*   **設計機制**：
    *   連續打卡 3 天：額外獲得 **+10% EXP**
    *   連續 7 天：額外獲得 **+25% EXP**，神獸獲得 `🔥 修行狂熱` 視覺效果。
    *   連續 14 天：額外獲得 **+50% EXP**，產生烈焰光暈。
*   **優化原因**：利用心理學的「損失規避（Loss Aversion）」，學員會因為害怕中斷連續記錄、失去高額加成，而形成每日必登入打卡的制約習慣。

## ⚡ 第二部分：系統架構與效能優化建議與原因

當系統實際開班、人數增加到數百人、打卡筆數破千後，現有架構會遇到嚴重的效能瓶頸。以下為優化方向與背後原因：

### 1. 全表無篩選平行查詢（資料庫瓶頸）
*   **現狀問題**：在前端加載資料時，系統平行撈取了 25 個資料表。然而，像是 `submissions`（打卡紀錄）與 `score_logs`（積分日誌）等表，是直接使用 `.select('*')` **無條件撈取全表歷史數據**。
*   **優化原因**：當期數累積、學員數量與每日打卡變多後（例如達數萬筆），每次使用者打開網頁或打卡，瀏覽器都要下載幾百 MB 的無用資料。這會導致網頁開啟極慢、白畫面時間長，且 Supabase 流量費用將急劇上升。
*   **優化方案**：
    *   限制查詢範圍：查詢 `submissions` 時，一律加上當前班期篩選：`.eq('batch_id', currentBatchId)`。
    *   對於積分日誌（`score_logs`），只加載當前用戶個人資料：`.eq('student_id', currentUser.id)`。

### 2. Client 端 $O(N^2)$ 雙迴圈關聯聯立（瀏覽器端瓶頸）
*   **現狀問題**：程式碼在 JavaScript 中手動將打卡紀錄與學員資料聯立，使用了大量陣列的 `.find()` 與 `.filter()`（雙重迴圈，時間複雜度為 $O(N^2)$）。
*   **優化原因**：當資料量達到千筆以上，雙重迴圈會導致瀏覽器主線程出現明顯的「凍結（Freeze）」，使介面動畫掉幀，使用者點擊按鈕時感覺「不靈敏」。
*   **優化方案**：
    *   改用 Map 進行 $O(1)$ 常數時間查詢：
        ```typescript
        const profileMap = new Map(profilesList.map(p => [p.id, p]));
        const joinedSubs = subsList.map(sub => ({
          ...sub,
          profile: profileMap.get(sub.student_id)
        }));
        ```
    *   改用 Supabase 原生 Join 語意，將聯立運算完全交給高效能的 Postgres 資料庫：
        ```typescript
        supabase.from('submissions').select(`
          *,
          profile:profiles(name, phone),
          mission:missions(title, points)
        `).eq('batch_id', currentBatchId)
        ```

### 3. 全局狀態變更引發全頁面重渲染（React 渲染瓶頸）
*   **現狀問題**：核心邏輯和上千行狀態都集中在單一組件中。只要刷新資料，無論學員目前停留在哪個分頁（排行榜、每日定課、後台），所有分頁組件都會同時觸發重渲染。
*   **優化原因**：切換頁籤或點選按鈕時，由於無效的 UI 計算過多，會產生明顯的遲滯感。
*   **優化方案**：
    *   使用 `React.memo` 包裝不常變動的重度渲染組件（如 `LeaderboardTab` 排行榜與 `AdminDashboard` 後台）。
    *   將寵物階段計算（`getActiveStage`）等高複雜度計算包在 `useMemo` 中，僅在依賴資料確實變動時才重新執行。

### 4. 資料庫索引優化（Supabase SQL 索引）
*   **優化原因**：隨著資料庫筆數增加，Postgres 資料庫在執行 Filter 或 Join 時速度會變慢。
*   **優化方案**：在 Supabase SQL 編輯器中為常用外鍵建立索引（Index），這能讓查詢速度提升 10 倍以上：
    ```sql
    CREATE INDEX IF NOT EXISTS idx_profiles_batch_id ON public.profiles(batch_id);
    CREATE INDEX IF NOT EXISTS idx_profiles_team_id ON public.profiles(team_id);
    CREATE INDEX IF NOT EXISTS idx_submissions_student_id ON public.submissions(student_id);
    ```
