# 建議書：學員跨期與多期同時參與之設定與顯示規劃

本建議書旨在解答「當學員同時參與兩個（或以上）期數時，系統應如何設定、顯示，以及如何處理資料關聯與安全權限」的問題。

---

## 一、 核心底層設計：資料庫模型

系統的資料表設計本質上**已經原生支援多期參與**：

1. **報名獨立性**：
   在資料庫中，`profiles` 表的一列代表一個學員的「一次報名/參與 (Enrollment)」。
   - 如果同一個學員同時參與 **49 期** 與 **50 期**，資料庫中會有 **兩筆獨立的 `profiles` 記錄**。
   - 這兩筆記錄會有不同的 `id`（報名ID）與 `batch_id`（期數ID），但會擁有相同的 `phone`（手機號碼）、`name`（姓名）以及 `profile_id`（個人識別ID）。

2. **資料歸屬**：
   學員的打卡記錄 (`submissions`)、分數明細 (`score_logs`)、修為與角色，都是綁定在特定的 `profiles.id` 上。
   - 這代表該學員在 49 期與 50 期的分數、打卡進度、所屬小隊、神獸等級是**完全獨立、互不干擾的**。這符合分期競賽與教學的商業邏輯。

---

## 二、 前端顯示與切換體驗

前端系統在 Header 中**已經具備期數切換邏輯**：

1. **自動識別**：
   學員登入後，系統會透過 `fetchData` 載入與該學員手機號碼相同的所有 `profiles` 記錄（存於 `userEnrollments` 狀態中）。
   
2. **無縫切換器**：
   如果 `userEnrollments` 的長度大於 1，系統會在 Header 右上方自動呈現**期數切換選單**。學員點擊切換後：
   - 系統會更新 localStorage 的 `nlp_mock_user_id` 為新期數的 `profile.id`。
   - 頁面會自動重新加載，展示該學員在該期數對應的小隊面板、每日打卡進度與神獸狀態。

> [!NOTE]
> 顯示部分的基礎建設已 100% 完備，無須額外開發。

## 三、 後台設定與指派方式

在後台要如何讓一個學員加入多期？我們提供兩種方案：

### 方案一：免改程式碼方案（推薦，即刻啟用）
* **操作方式**：
  大隊長在後台建立好新期數與小隊後，將**該小隊的邀請連結或二維碼**直接發送給學員。學員點擊連結，在註冊頁輸入與先前**完全相同**的姓名與手機號碼。
* **底層行為**：
  註冊邏輯會偵測到這是既有學員，並在 `profiles` 中新增一筆綁定新期數的 profile 記錄。
* **優點**：
  - **零開發成本、零風險**，完全不需要修改任何程式碼。
  - 大隊長與學員的操作直覺，由學員主動掃碼加入新期。

### 方案二：後台指派方案（需修改後台 UI 程式）
* **操作方式**：
  在後台「學員/人事管理」新增一個「**指派既有成員到此期**」的按鈕。大隊長點擊後，從去重後的歷史成員下拉清單中選擇該學員，並為其指定新期數的小隊、角色與狀態。
* **底層行為**：
  在後台 insert 一筆新 `profiles`，拷貝其姓名、手機、`profile_id`，並寫入新的 `batch_id`、`team_id` 與 `role`（他可以在 A 期是學員，B 期指派為小隊長）。
* **優點**：
  大隊長可完全主控，不需要學員掃碼。
* **缺點**：
  需要花費 0.5 ~ 1 天時間修改 `AdminDashboard` 後台元件，且會增加後台複雜度。

## 四、 安全防線 (RLS & JWT) 的配合

隨著 Tier 3 安全鎖通電，多期學員的權限同樣安全無虞：

1. **唯一 Auth 帳號**：
   在 `/api/auth/login` 新登入流程中，學員使用手機號碼登入時，系統會將該手機號碼綁定至 Supabase Auth 的同一個使用者 UUID (`auth.uid()`)。
   
2. **自動連鎖綁定**：
   登入時，後端會自動將該手機號碼對應的**所有期別 profiles** 的 `auth_user_id` 欄位更新為同一個 Auth UUID。
   
3. **安全讀寫**：
   學員無論在前端切換到哪一期，其發送的 JWT Token 都會對應同一個 `auth.uid()`。資料庫的 RLS 政策會因為 `auth_user_id = auth.uid()` 而允許他讀寫自己這兩期的所有打卡與神獸資料，防竄改保護完全適用。

## 五、 常見問題 (FAQ)

### Q：大隊長在後台「學員名單」中，會看到同名的學員出現兩次嗎？
**會的，但這在業務邏輯上是正確的。**

1. **篩選「所有期數」時**：
   - 學員「王小明」若同時參加 49 期與 50 期，在後台列表選擇「所有期數」時，確實會看到兩列「王小明」。
   - **為什麼這是對的？** 因為這是兩次獨立的「報名 (Enrollment)」。其中一列顯示的期數是 49 期（例如：角色是學員，分數 250 分，狀態已結業）；另一列顯示 50 期（例如：角色是小隊長，分數 0 分，狀態使用中）。大隊長需要能獨立編輯這兩筆資料（如調整分數、分配小隊）。

2. **篩選「特定期數」時**：
   - 如果大隊長在後台下拉選單過濾「只顯示第 50 期」，則只會出現該學員在第 50 期的那一列資料，畫面依然會非常乾淨。

## 六、 行動建議與決策

1. **初期運營建議**：
   直接使用 **方案一**（學員掃描新期數邀請碼註冊）。這不需要寫任何程式，且完全相容於目前的 RLS 與 Header 期數切換器，是立即可用且最穩定的做法。
   
2. **中長期優化建議**：
   在本次「程式碼上正式線」穩定運行後，若大隊長強烈要求「不想讓學員掃碼，想直接指派」，我們再於後台 `RosterTab` 擴充方案二的 UI 設定。
