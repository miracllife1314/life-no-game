# 階段 3：正式區安全遷移計畫書

> 目的：把測試庫驗收過的安全（真實登入 + RLS + 欄位保護 + 限流）搬到**正式站**，保護真實學員。
> 原則：**先部署程式（不開 RLS，零風險）→ 給學員重新登入的緩衝 → 離峰時才開 RLS**。全程可一鍵回退。
> ⚠️ 本文是計畫，尚未執行。每一步要動正式區前都會再跟你確認。

---

## 🔑 最關鍵的一件事（先懂這個，其他才看得懂）

開了 RLS 之後，資料庫會用「**你登入身分(auth.uid)**」判斷能不能讀寫。
- 學員**用新流程登入過一次**＝拿到真實身分 → 一切正常。
- 學員**還在用舊的自動登入**（沒有新身分）＝ RLS 會把他擋住 → 打卡/看分數會壞。

所以整個遷移的核心 = **讓每個還在上課的學員，在開 RLS 之前，至少用新版登入過一次。**
這就是為什麼要「先部署程式 + 公告重新登入 + 緩衝幾天」，最後才開 RLS。

---

## 1. 正式區備份流程（動任何東西前必做）

1. 本機 `.env.local` 要有正式區的無後綴變數（URL/anon）與 `SUPABASE_SERVICE_ROLE_KEY`。
2. 執行：`npm run backup --prod` → 會把正式庫整個備份到 `backups/日期_時間/`，並印出每表筆數。
3. **核對筆數**合理（profiles/missions/submissions… 數字對得上線上現況）。
4. 雙保險：Supabase Pro 本身有「**每日自動備份、保留 7 天**」，必要時可在後台一鍵還原到某個時間點。

> 沒備份成功、沒核對筆數，就**不要**往下做。

---

## 2. 正式區 service role 設定

後端核發身分（`/api/auth/login`、限流）需要正式區的 service role 金鑰。
1. Supabase **正式專案**（`epolsiukauqfwxmjojia`）→ Settings → API Keys → **Secret key（service_role）** 複製。
2. 到 **Vercel → 專案 → Settings → Environment Variables → Production**：
   - 新增 **`SUPABASE_SERVICE_ROLE_KEY`**（**無 `_LOCAL` 後綴**）= 剛複製的金鑰。
   - 確認 Production 仍是無後綴的 `NEXT_PUBLIC_SUPABASE_URL` / `ANON_KEY`（指向正式庫）。
   - ⚠️ **Production 絕不可出現任何 `_LOCAL` 變數**（否則正式站會連到測試庫）。
3. 程式讀法：`SUPABASE_SERVICE_ROLE_KEY_LOCAL || SUPABASE_SERVICE_ROLE_KEY` → 正式區自動吃無後綴那把。
4. 不需要開 Phone/簡訊 provider（我們用 generateLink，不發簡訊）。

---

## 3. 正式區 SQL 執行順序

我會準備**一支整合好的「正式區遷移 SQL」**（內容＝測試庫驗收過的那套）。執行順序：
1. **加欄位**：`profiles` 新增 `auth_user_id`（+索引）。
2. **限流表**：建 `login_attempts`（鎖死，只 service role 用）。
3. **安全主體**（＝ `FIX_security_recover.sql` 的內容）：
   輔助函式 → 3 個 INVOKER 欄位保護 trigger → `adjust_score` 守門 →
   清舊政策 → 建嚴格 RLS 政策 → **最後才 `ENABLE RLS`（profiles/submissions/score_logs/student_notes/teams）**。

> 重點：**第 3 步的「ENABLE RLS」是整個遷移真正『通電』的那一刻**——前面 1、2 步先做完、放著都不影響線上（只是加欄位/加表）。所以可拆成兩天：先做 1、2 + 部署程式，最後一天才做第 3 步。

---

## 4. 正式站部署流程

1. 把 `staging` 的程式合併到 `main`（或部署目前這版到 Production）→ 觸發 Vercel 正式建置。
2. **先部署程式、但先不開 RLS**（SQL 第 3 步的 ENABLE 還沒跑）。此時：
   - 新版登入流程上線 → 學員登入會**自動拿到真實身分並綁定**，但**體驗完全不變**（一樣打姓名+電話）。
   - 因為 RLS 還沒開，沒登入新流程的人也照常用 → **零風險**。
3. 建議**先加一個小程式保護**（我會做）：開機時若偵測不到有效新身分，就**自動帶回登入頁**（而不是用舊的自動登入硬進去）。這樣等開了 RLS，沒人會卡在「進得去但讀不到」的壞狀態。

---

## 5. 學員重新登入公告方式

1. 程式部署後（RLS 還沒開），發公告：
   - App 內公告 +（LINE/群組）：「**系統升級完成，請各位重新登入一次以套用新版**，操作跟以前一樣，輸入姓名+電話即可。」
2. 緩衝 **3～7 天**，讓大家自然 + 看到公告後陸續重新登入（＝拿到新身分）。
3. （可選，更穩）我寫一支**預先綁定腳本**：用 service role 幫所有現有學員先建立 auth 身分、回填 `auth_user_id`。這樣他們登入時是「秒綁定」，更不容易出錯。
4. 開 RLS 前，我可以查「**已綁定比例**」（多少 profiles 有 auth_user_id），確認覆蓋夠高再通電。

---

## 6. 出錯時如何 Rollback（隨時可秒退）

| 狀況 | 處置 |
|---|---|
| 開 RLS 後有合法功能被擋 | 跑 **ROLLBACK SQL**：`DISABLE ROW LEVEL SECURITY`（5 表）+ drop 我們的 3 個 trigger → **立刻回到開放狀態**，App 照常。 |
| 新版程式有問題 | Vercel **一鍵 rollback** 回上一個部署。 |
| 資料被改壞 | `npm run restore --prod 備份名 CONFIRM`，或 Supabase 後台還原到某時間點。 |
| 登入核發身分失效 | 設 Vercel 環境變數 `NEXT_PUBLIC_USE_REAL_AUTH=false` → 程式退回舊「姓名+電話」登入。 |

> ROLLBACK SQL 我會事先準備好、存成 Supabase 的 saved query，出事一鍵貼上跑。

## 7. 上線前我（你）要手動測哪些流程

部署後（先在 RLS 關著時測一輪，開 RLS 後再測一輪）：

**學員**：重新登入 → 每日打卡（自動通過的）→ 需審核任務打卡 → 看分數/排行榜 → 看寵物（神獸在中間）→ 見證牆瀏覽 + 發一篇分享。
**隊長**：登入 → 審核/退回隊員打卡 → 指揮所手動幫隊員打卡 → 改自己隊設定（邀請碼/人數）→ 指派小隊角色。
**大隊長(gm)**：登入 → 手動調分 → 新增/編輯學員 → 換隊 → 發任務 / 刪任務 → 課程出席+1000 → 檢視某學員(代打卡)。
**安全**（開 RLS 後）：在 F12 主控台試「改自己分數 99999」「偽造打卡」→ 應全部被擋。

> 我也會準備一份「正式區版」的攻擊檢查，由我在你監督下對正式站跑一次確認 6 項全擋（或你手動 F12 測）。

## 8. 適合上線（開 RLS）的時間點

- **離峰、學員最少**的時段：通常是**深夜**（當天打卡都結束後），或課程之間的空檔日。
- **避開**：每日打卡截止前、上課日的活躍時段、任何活動/抽獎當下。
- 採**兩階段**降風險：
  - 階段 A（任意時間，低風險）：備份 + 部署程式 + 加欄位/限流表 + 發公告。
  - 階段 B（離峰、緩衝幾天後）：**開 RLS（通電）** + 立刻手動測一輪 + 守著監看 30～60 分鐘，沒事才安心。
- 開 RLS 後**留 1～2 小時**在電腦前，萬一有人反映進不去可立刻 rollback。

```
              有事 → 一鍵 ROLLBACK
```
