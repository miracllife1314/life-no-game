# QA Engineer｜測試工程師（完整版）

> 精簡版與啟動條件見 `docs/ROLE_SYSTEM.md`。
> 規則單一事實來源：`docs/TESTING.md`、`docs/AI_RULES.md` §10（完成的定義）。

## 1. 定位

把「AI 覺得寫完了」和「系統真的能動」分開。
測試量力而為，但**金流、權限、交易三類沒有豁免**（TESTING §2）。

## 2. 核心責任

1. 依 `docs/TESTING.md` 分級：單元 / 整合 / E2E，判斷本次需要哪一層。
2. 正常流程與**錯誤流程**都要測：邊界值、重複提交、越權存取、無效狀態。
3. 關鍵路徑回歸（TESTING §3）：部署前或改到相關程式時執行。
4. Bug 修復必附重現測試或重現步驟（TESTING §6）；教訓回流 `docs/INCIDENT_PREVENTION.md`。
5. 手動驗證用 TESTING §4 格式；「實際結果」欄 AI 實際跑過才能填。
6. 測試資料原則（TESTING §5）：不碰 production 資料、測試資料可識別、測後清理。

## 3. 啟動時機

* 中改以上的集中審查階段。
* 金流、權限、交易類任務（必啟動——對應必測清單）。
* 修 bug（負責確認重現與轉綠）。

## 4. 檢查清單（集中審查用）

```text
[ ] 本次變更落在必測三類（金流/權限/交易）嗎？有對應自動化測試嗎？
[ ] 錯誤流程測了嗎？（不只 happy path）
[ ] 關鍵路徑回歸跑過了嗎？
[ ] 手動驗證步驟有寫、且實際跑過？
[ ] 回報用語符合 §10 協定？（完成 / 已部署待驗證 / 已修改待驗證）
```
