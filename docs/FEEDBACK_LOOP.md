# FEEDBACK LOOP — 模板回饋循環

> 目的:讓「真實開發中發現的問題」能有系統地回流,持續打磨母版模板。
> 這是模板從「設計良好」走向「實戰驗證良好」的關鍵機制。

## 為什麼需要

模板是**複製**到各專案使用的。在某個專案裡發現「這條規則太嚴 / 少一條防範 / preflight 有 bug」,如果不回流,母版永遠學不到,每個新專案都會再踩一次同樣的坑。

## 三個階段

### 階段一:察覺(開發中)

出現以下任一訊號,就是回饋——記到 `logs/template_feedback.md`:

| 訊號 | 代表 |
|---|---|
| AI 為小事停下來問你 | 規則太嚴 |
| AI 沒問就做了危險的事 | 規則太鬆或缺漏 |
| 你很想跳過檢查（--no-verify / 叫 AI 別管規則） | 規則不可行 |
| preflight 誤報或漏報 | 工具要修 |
| 你重複糾正 AI 同一件事（>2 次） | 該固化成規則 |
| bug 溜到你面前、但它本該被某規則擋住 | 缺一條 INCIDENT_PREVENTION |
| AI 讀錯文件 / 依過期資訊判斷 | 文件過期或雙頭馬車 |

**最有價值的是「重複糾正」和「本該擋住的 bug」**——這兩個是把個人經驗變成規則的金礦。

### 階段二:收集(收工時)

每完成一個功能或收工前,執行 `logs/template_feedback.md` 裡的 **Retro Prompt**,
讓 AI 自動把當次的回饋整理進該檔。你不用主動記,交給 AI。

### 階段三:回流(週期性)

每完成一個大功能、或每週一次,把 `logs/template_feedback.md`
帶回「萬用AI系統開發2版」母版對話,由母版判斷:

```text
每一條回饋 →
  · 通用（別的專案也會遇到）→ 改母版 + 升版本號 + 記 TEMPLATE_CHANGELOG
  · 只是本專案特例          → 留在專案，不污染母版
回流後，把該筆標「是否已回流母版：是」
```

其他既有專案要不要拉母版新版,看 `docs/TEMPLATE_CHANGELOG.md` 自行決定
(對照各專案頂部標註的「基於模板 vX.X.X」)。

**Append-only 原則**:feedback 是歷史紀錄。整併或回流時一律**新增**「整併結論 / 決議」條目,
不得覆蓋或刪除舊紀錄——舊紀錄是判斷「同一問題是否重複發生」的證據。

## 規則預算（防規範膨脹）

實戰已證明：**文字規則會衰減，機器檢查會存活**（v1.3.2 教訓）。因此回流新增規則前，依序問：

1. **能不能做成機器檢查**（preflight / hook / 測試 / DB 約束）？能 → 優先機器化，文字只留一句指路。
   （原則同 `docs/AI_RULES.md` §0.1「規則優先機制化」——該節為原則 SSOT，本點只是回流關卡的落地，不另立規則。）
2. 無法機器化的文字規則，回流時須寫明「為何無法機器化」才進母版。
3. 優先**替換或整併既有規則**，而非純疊加——母版總量成長要有意識，不是無上限。

## 判斷「通用 vs 特例」的準則

- **通用**:換一個專案、換一個業務,這個問題還會發生嗎?會 → 進母版。
  例:preflight 大小寫盲點、文件雙頭馬車——任何專案都會遇到。
- **特例**:只跟這個專案的特定資料表 / 特定業務有關 → 留專案。
  例:某張特定表的欄位命名習慣。

## 已驗證的實例

模板多個版本全部來自真實專案回流:
- v1.1.1 preflight 註解誤報 —— brownfield 匯入時發現
- v1.1.2 preflight 大小寫盲點 + inline 白名單 —— 協作 AI 盤點發現
- v1.1.3 單一事實來源規則 —— 兩份 AI_RULES 並存時固化
- v1.2.1 外部白名單檔 —— 發現 inline 白名單無法用於不可變 migration（回饋循環第二圈）
- v1.3.2 回報用語協定 + 一功能一 commit + 金流無測試提醒 —— **第一次透過 template_feedback.md 正式回流**；核心發現：文字規則會衰減（DoD 被放寬、TESTING 被跳過），機器檢查會存活（preflight 每次 commit 都有跑）→ 之後的修正優先做「機器化」與「具體到無法打折的用語」
- v1.3.3 快慢分層 —— **兩則對立回饋（要擋 Vercel 靜默失敗 vs 別拖慢 commit）整合成一條規則**：快檢進 pre-commit、完整 build 進 pre-push。示範回流的另一個要點：**同議題的多則回饋必須整合，不得各寫一條規則**（否則規則自己變雙頭馬車）
- v1.7.0 一週使用率盤點回流 —— **第一次「負向證據」回流**：不只補規則，還依 0% 執行率**裁撤**規則（🛡️ 全域宣告退役、角色開會制退役、docs/roles/ 降級），並依使用者裁決**否決**一項建議（R7 行數提醒，記 FUTURE_BACKLOG 候選）。示範回流的第三個要點：**回流也要做減法與說不**——執行率是規則的生死判準，留著沒人執行的規則會損傷整套規則的權威

這證明這條循環是有效的——**用真實專案磨模板,比憑空加規則有價值得多**。
特別是 v1.2.1：它修正的正是 v1.1.2 自己引入的設計矛盾，說明「回流」不只補漏，也會揪出前一次改進的盲點——循環要持續轉，不是轉一圈就結束。
