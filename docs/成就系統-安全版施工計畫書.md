# 遊戲化成就系統 — 安全版施工計畫書

> 把「成就系統 V3 設計規格」轉成**可安全施工**的計畫。重點:分階段、先測試庫/測試站、每步驗證、不破壞現有計分與安全鎖。

---

## 0. 動工前必讀:4 個會「上線當天爆炸」的風險(已內建修正)

| # | 風險 | 後果 | 修正(本計畫已納入) |
|---|---|---|---|
| 1 | `user_achievements.notified` 預設 false | 上線當天**每個學員被歷史成就彈窗洗版**(一次跳 10+ 張) | 加欄位後**立即回填** `notified=true`,只有之後新解鎖才彈 |
| 2 | 前端標記 `notified=true` = 學員寫 `user_achievements` | 與今天的 RLS(該表**只准管理員寫**)衝突 → **被擋、彈窗永遠關不掉** | **不放寬 RLS**;改用 SECURITY DEFINER RPC `mark_achievements_notified()` |
| 3 | 成就檢查要改 `_submission_score_trigger` | 那支**剛改成 +200 見證獎勵**;若從舊版改會**洗掉 +200** | 一切**接在目前最新(+200)版本**上,先抓現行函式定義再改 |
| 4 | `consecutive_checkins`(連勝)在 SQL trigger 算 | 複雜易錯、觸發時機不對 | **最後做或複用前端 streak**;前期先上簡單條件 |

> ⚠️ **鐵律**:任何動到「計分 trigger / RLS」的 SQL,**先在測試庫跑 + 行為驗證,再上正式庫**。動正式庫前先在 Supabase 後台做一次備份。

### 0.5 🔴 核心原則:不溯及過往(跟見證 +200 同精神)

**意思**:過去的行為**不補發成就、不彈窗**;成就**從上線那一刻起才開始算**,只有「上線後新發生的」才會解鎖+彈窗。
(就像 +200:舊的 +300 不變,只有新入選才 +200。)

**為什麼非處理不可**:階段 C 的 trigger 預設會拿「**全部歷史**」比對條件。若不處理,老學員下次一打卡就會**瞬間解鎖一堆過去就達標的成就 + 彈窗洗版**——正是要避免的「溯及既往」。

**做法(上線時跑一次「基準線」)**:把每位學員**現在就已達標**的成就,**直接補成已解鎖且 `notified=true`(不彈窗)**,當作起跑基準。之後 trigger 只會解「上線後才新達標」的(`notified=false` → 彈窗)。

```sql
-- 上線基準線:現在已達標的全部標記為「已解鎖+已通知」,當起點,不彈窗、不溯及。
-- (各 condition_type 依階段 C 的條件分別 insert;統一 notified=true)
-- 範例(total_score 型;其餘型別比照階段 C 的條件):
insert into public.user_achievements (student_id, achievement_id, unlocked_at, notified)
select p.id, a.id, now(), true
  from public.profiles p
  join public.achievements a
    on a.condition_type='total_score' and a.condition_value <= coalesce(p.score,0)
 where not exists (select 1 from public.user_achievements ua
                    where ua.student_id=p.id and ua.achievement_id=a.id);
-- mission_count / witness_post_count / pet_stage 各跑一段(條件同階段 C，notified 一律 true)
```

> 兩種選擇,本計畫採**第 1 種**:
> 1. **(採用)補成已解鎖但不彈窗**:徽章反映真實成就、但過去不洗版。最符合「不溯過往的彈窗」且公平。
> 2. (備選)過去完全不算:count 型條件改成「只數上線後(`created_at >= 上線時間`)的紀錄」——較複雜,且老學員會覺得「我之前做的都不算」。除非你明確要這種,否則用第 1 種。

## 1. 資料模型現況(動工前先確認)

- `achievements`:`id, title, description, icon_url, condition_type, condition_value`(目前 `condition_type` 只用到 `total_score`)。
- `user_achievements`:`id, student_id, achievement_id, unlocked_at`。
- 現行自動解鎖:`_apply_score_delta()` 的 2e 段已會解 `total_score` 型成就(`condition_value <= 新分數`)。
- RLS(今天上的):`achievements` / `user_achievements` 皆 **公開讀 + 只有 `is_admin()` 能寫**;計分 trigger 是 SECURITY DEFINER → 繞過 RLS。
- 計分 trigger 最新版 = 見證 +200(`docs/schema_fixes_11_witness_bonus_200.sql`)。

## 階段 A — 資料庫地基(低風險,先做)

**做什麼**:加欄位 + 回填 + 通知 RPC。**不碰 trigger、不碰計分。**

```sql
-- A1. achievements:任務綁定欄位(mission_count 型用);condition_type 維持 text,新值由應用層寫入
alter table public.achievements add column if not exists target_mission_id text;

-- A2. user_achievements:通知旗標
alter table public.user_achievements add column if not exists notified boolean not null default false;

-- A3. 🔴 關鍵回填:歷史成就一律視為「已通知」,避免上線洗版
update public.user_achievements set notified = true where notified = false;

-- A4. 通知 RPC(SECURITY DEFINER → 繞過「只准管理員寫」的 RLS;學員前端呼叫它來關彈窗)
create or replace function public.mark_achievements_notified(p_student_id text)
returns void language plpgsql security definer set search_path = public as $$
begin
  update public.user_achievements
     set notified = true
   where student_id = p_student_id and notified = false;
end; $$;
grant execute on function public.mark_achievements_notified(text) to anon, authenticated;
```

**驗證(測試庫)**:
- `select count(*) from user_achievements where notified=false;` → 回填後應為 **0**。
- 呼叫 `select mark_achievements_notified('<某測試學員id>');` 不報錯。
- 確認 `achievements` 多了 `target_mission_id`、`user_achievements` 多了 `notified`。

**部署關卡**:測試庫驗證 OK → 再貼正式庫。

## 階段 C — trigger 擴充「簡單條件」(中高風險:動計分 trigger,務必小心)

**做什麼**:讓解鎖支援 `mission_count` / `witness_post_count` / `pet_stage`(都是 count/比大小,好寫好測)。**先不做 consecutive_checkins。**

**做法(關鍵)**:
1. **先抓現行最新函式定義**(別憑記憶):
   ```sql
   select pg_get_functiondef('public._apply_score_delta'::regprocedure);
   select pg_get_functiondef('public._submission_score_trigger'::regprocedure);
   ```
2. 新增一支集中的解鎖檢查 helper,把它**加進(不是取代)** `_apply_score_delta` 結尾(取代原本只檢查 total_score 的 2e 段):
   ```sql
   create or replace function public._check_unlock_achievements(p_student_id text, p_score integer)
   returns void language plpgsql security definer set search_path = public as $$
   begin
     -- total_score
     insert into public.user_achievements (student_id, achievement_id, unlocked_at)
     select p_student_id, a.id, now() from public.achievements a
      where a.condition_type='total_score' and a.condition_value <= coalesce(p_score,0)
        and not exists (select 1 from public.user_achievements ua where ua.student_id=p_student_id and ua.achievement_id=a.id);

     -- mission_count(完成某任務 N 次:approved 計數)
     insert into public.user_achievements (student_id, achievement_id, unlocked_at)
     select p_student_id, a.id, now() from public.achievements a
      where a.condition_type='mission_count' and a.target_mission_id is not null
        and (select count(*) from public.submissions s
              where s.student_id=p_student_id and s.mission_id=a.target_mission_id and s.status='approved') >= a.condition_value
        and not exists (select 1 from public.user_achievements ua where ua.student_id=p_student_id and ua.achievement_id=a.id);

     -- witness_post_count(入選見證牆 N 則)
     insert into public.user_achievements (student_id, achievement_id, unlocked_at)
     select p_student_id, a.id, now() from public.achievements a
      where a.condition_type='witness_post_count'
        and (select count(*) from public.submissions s
              where s.student_id=p_student_id and s.status='approved' and s.share_to_witness=true and s.mission_id<>'task-custom-post') >= a.condition_value
        and not exists (select 1 from public.user_achievements ua where ua.student_id=p_student_id and ua.achievement_id=a.id);

     -- pet_stage(神獸階段 >= N)
     insert into public.user_achievements (student_id, achievement_id, unlocked_at)
     select p_student_id, a.id, now() from public.achievements a
      where a.condition_type='pet_stage'
        and exists (select 1 from public.user_pets up where up.student_id=p_student_id and coalesce(up.current_stage_index,0) >= a.condition_value)
        and not exists (select 1 from public.user_achievements ua where ua.student_id=p_student_id and ua.achievement_id=a.id);
   end; $$;
   ```
   然後在 `_apply_score_delta` 末尾把原本的 2e 段**換成** `perform public._check_unlock_achievements(p_student_id, v_new_score);`
3. **`_submission_score_trigger` 維持 +200 版不動**(它本來就會呼叫 `_apply_score_delta`,自然帶到新解鎖邏輯)。

**驗證(測試庫,務必)**:
- 建測試成就(各類型各一)→ 用 service role 模擬一筆 approved 打卡/見證 → 查 `user_achievements` 是否正確新增、**且 +200 仍正常、基礎分數沒亂**(用我之前那種「插入→看 score_logs→刪除還原」的腳本)。
- 跑 `npm run test-flows` + `npm run security-check`。

**部署關卡**:測試庫全綠 + 行為驗證過 → **備份正式庫** → 才上正式庫。

## 階段 D — 前端解鎖彈窗佇列(中風險:純前端)

**做什麼**:
- 載入 `user_achievements` 後,把 `notified=false` 的推進 `pendingAchievements` 佇列。
- 一次彈一張霓虹賀卡 + 五彩噴砂;點「收下修煉祝福」→ 呼叫 **`mark_achievements_notified(student_id)`**(階段 A 的 RPC)→ 本地把該張移出佇列 → 自動彈下一張。
- **不要**用 `supabase.from('user_achievements').update(...)`(會被 RLS 擋)——一律走 RPC。

**注意 SWR**:我們有 SWR 快取。彈窗判斷要用「最新撈回」的 user_achievements,避免拿快取舊資料重複彈(可在 fetch 完成後才觸發佇列)。

**驗證**:測試站新解一個成就 → 彈一張 → 點收下 → 不再彈、重整也不再彈(notified 已寫回)。

## 階段 E — 連勝型成就 consecutive_checkins(最後做)

**問題**:SQL trigger 算「連續不重複日期」複雜易錯,且我已在前端做了「連續修行天數 🔥」(`DailyQuestsTab` 的 `dailyStreak`)。

**建議二選一**:
- (a) **前端判斷**:在算出 `dailyStreak` 後,若達某成就門檻就呼叫一支 `unlock_achievement_if(student_id, achievement_id)` RPC(SECURITY DEFINER,內含 `not exists` 防重複)。簡單、複用既有邏輯。
- (b) 若堅持後端算:寫一支獨立函式用 `generate_series`/視窗算連續日期,**單獨充分測試**再接上。

**先把 A~D 上線穩定,E 再說。**
