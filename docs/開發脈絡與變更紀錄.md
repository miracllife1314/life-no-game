## Part 1 — 檢查前必讀的重要脈絡 / 陷阱

### 🗄️ 資料庫:有兩個
- `_LOCAL` 變數 = **測試庫**(本機 dev、Vercel Preview/staging 用)。
- 無後綴變數 = **正式庫**(Vercel Production / `main`,學員正在使用)。
- `lib/supabase.ts`:只要有設 `_LOCAL` 就用測試庫,否則用正式庫。
- **正式庫絕不可設 `_LOCAL` 變數**。

### 👤 profile_id 的陷阱(G 上次誤判的點)
- 一個「人」跨多期會有多筆 profiles,各筆 `id` 不同、但用 `profile_id` 串起來。
- **`app/page.tsx` 約 224 行:`const profile_id = p.profile_id || p.id`** → 前端拿到的 profiles,`profile_id` **永遠不會是 null**(null 會被補成 `id`)。
- 所以「手動新增的帳號 profile_id 在 DB 是 null」**不代表前端會出錯**,別用「DB 是 null」推論前端會壞。

### 🔑 captain_id 與 director_id「比對的鍵不同」(很容易搞錯)
- **captain_id**(隊長):用 **`profile_id`** 比對。例:`profiles.find(p => p.profile_id === team.captain_id)`。隊長候選下拉 value 綁 `c.profile_id`(正確)。
- **director_id**(大隊長):用 **`profiles.id`** 比對。例:`profiles.find(p => p.id === director_id)`(CaptainDashboard、TeamsTab)。大隊長下拉 value 綁 `p.id`(2026-06 修正,見 Part 2)。
- **改下拉前先確認這個欄位是用 id 還是 profile_id 比對**,不要一律改成同一個。

### 🧮 計分的真相在哪
- **分數真相 = `profiles.score`**,由資料庫 trigger(`schema_fixes_11`)原子更新。
- `score_logs` **只是顯示歷史**,不參與計分。少載/延後載不影響分數正確性。
- 審核通過給分:trigger 依 `submissions.score_awarded` 加分;上見證牆額外 +200。退回則扣回。

### ⏱️ 大隊長(admin)載入:延後載入策略(services/queries.ts)
- `fetchFull`(admin)所有查詢是 **`Promise.all` 並行** → **登入時間 = 最慢那一張查詢**。砍掉非最慢的表不會更快。
- **`score_logs` 延後載入**:登入「不載」,大隊長切到 `history` / `captain` 分頁時才載一次(`app/page.tsx` 的 `loadScoreLogs`)。所以 admin 一登入 `scoreLogs` 是空的,是**正常的**,別當成 bug。
- `applyTables` 用 `if (scoreLogsList.length > 0)` 才覆蓋,避免空陣列把已載入的歷程蓋掉。
- 學員/隊長走 `fetchScoped`,自帶本期歷程,不受上面影響。

### 🖼️ 證明圖片(proof_image_url)
- 曾有 11 筆是 **base64 整張圖**塞在欄位裡(撐到 ~10MB,害登入超慢)。
- `lib/supabase.ts` `uploadProofImage`:上傳 Storage 失敗會重試,**一律不退回 base64**(回 null),DB 不會再被撐肥。
- `fetchFull` 撈圖時用 `.not('proof_image_url','like','data:%')` **排除 base64**,只載正常 URL 圖(很小)。那 11 筆舊 base64 圖暫時不顯示,待日後搬 Storage。

### 🔒 RLS 現況(影響你用 anon key 能不能讀/寫)
- `profiles` 已鎖(個資批①):anon 讀不到;公開查詢改用 **`v_public_profiles`** 視圖(無手機)。
- `submissions` / `teams` 仍 `using(true)`:anon 可讀。
- DDL/資料修正:**Claude 沒有正式庫寫入權限**,需使用者把 SQL 貼到 Supabase SQL Editor 執行(慣例見 `docs/schema_fixes_*.sql`)。

### 🧩 任務:範本 vs 任務本體(review_type 同步)
- `mission_templates`(後台「任務管理」顯示的)= 範本;`missions`(學員實際打卡)= 任務本體。
- 改範本的 `review_type` 後,既有 `missions` 會**自動同步**(`hooks/useAdminMisc.ts` `handleUpdateMissionTemplate`,2026-06 加)。歷史不一致資料用 `docs/schema_fixes_21` 對齊。
- `review_type`:`auto`(免審核,自動通過,`reviewed_by` 會是 null)/ `leader`(隊長可審)/ `admin`(只有大隊長可審)。

### 📊 PostgREST 預設 1000 筆上限(大表必踩)
- Supabase/PostgREST 查詢**沒設 `.limit()` 時,預設最多只回 1000 筆**,且**沒排序就是回「最舊的」**。
- `submissions` 在 2026-06-25 破 1000 筆 → 後台撈提交被截到最舊 1000、**最新待審全沒了**,造成「學員提交了、後台卻沒任務可審」。
- 規則:**會成長的大表(submissions/score_logs…)查詢一律 `.order('created_at',{ascending:false}).limit(大數字)`**;像「待審核」這種絕不能漏的,**另外用 `.eq('status','pending')` 單獨撈一次再合併**(見 `queries.ts` fetchFull/fetchScoped)。

### 🔗 巢狀關聯在 JS 端補(services/joinData.ts)
- PostgREST 不自動 embed,所以 `submissions` 的 `.mission` / `.profile` 是在 `computeJoinedData` 用 Map 補上的。
- 後台若有畫面顯示空白/未知,先確認該元件是「靠 join 的欄位」還是「自己用 id 反查 props」。ReviewsTab 已改成用 props 反查(較穩)。
