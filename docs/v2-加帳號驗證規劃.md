# v2 規劃：加入帳號驗證與資料庫權限（RLS）

> 目的：解決目前「資料庫對所有人完全開放」的問題，讓只有本人能改自己的資料、
> 只有管理員/隊長能做管理操作。**這份是施工藍圖，建議排在兩期課程中間的空檔做。**

---

## 0. 一句話講清楚現在的問題

系統用「姓名/手機」查表登入，**沒有真正的帳號**，所以資料庫分不出誰是誰，
只能把權限（RLS）全開（[schema.sql:352](schema.sql#L352)）。
要分權限，**前提是先有「帳號驗證」**——這就是 v2 的核心。

---

## 1. 要先決定的事：用哪種登入方式？

這是唯一需要你拍板的決定，其他都是工程。三個選項：

| 方式 | 優點 | 缺點 | 適合度 |
|---|---|---|---|
| **A. 手機簡訊驗證碼 (OTP)** | 最貼近現在的手機登入、學員無痛 | 簡訊要錢（每則約 NT$0.5–2，需接 Twilio） | ⭐ 推薦 |
| **B. Email + 密碼** | 免費 | 學員不一定有/會用 email、要記密碼 | 普通 |
| **C. 一組「課程通行碼」+ 姓名** | 最簡單、免費 | 安全性最弱（通行碼會外流） | 救急用 |

**建議 A**（簡訊驗證碼）：體驗最接近現狀，學員只要收個驗證碼。
費用很低（幾十個學員一期不到台幣幾十元）。

> 👉 你先選一個，剩下的我可以照這份藍圖實作。

## 2. 整體改動：前 vs 後

**現在：**
```
學員打開網頁 → 輸入姓名 → 查 profiles → 存 localStorage → 用 anon key 直接讀寫所有資料
                                                              （RLS 全開，誰都能改）
```

**v2 之後：**
```
學員打開網頁 → 輸入手機 → 收驗證碼 → 通過 Supabase Auth → 拿到「身分令牌」
                                                          → 讀寫時資料庫自動檢查身分
                                                            （只能動自己的；管理操作要管理員身分）
```

關鍵：每個人有一個**真實 auth 帳號**，資料庫每次讀寫都能確認「你是誰、能不能做這件事」。

---

## 3. 資料模型怎麼接（重要）

目前 `profiles` 一列 = 一筆報名；同一個人多期會有多列（共用 `profile_id`）。
所以一個 auth 帳號要對應到「**一個人**」（= 多筆報名列）。

做法：
1. `profiles` 新增欄位 `auth_user_id uuid`（指向 Supabase Auth 的使用者）。
2. 同一個人的所有報名列，填同一個 `auth_user_id`。
3. 管理員/隊長身分：建一個 `user_roles` 表（`auth_user_id`, `role`），
   由後台設定，**RLS 從這裡判斷你是不是管理員**（不能放在前端，會被竄改）。

---

## 4. RLS 政策設計（每類表怎麼設）

用一個輔助函式 `is_admin()`（查 `user_roles` 是否為 admin）來簡化。

| 資料表 | 讀取 (SELECT) | 新增/修改/刪除 |
|---|---|---|
| `profiles` | 本人可讀自己的；管理員可讀全部 | 本人可改有限欄位；管理員全權 |
| `submissions`（打卡） | 本人 + 同隊隊長 + 管理員 | 本人可新增自己的；審核/刪除限隊長/管理員 |
| `score_logs`（分數紀錄） | 本人可讀；管理員全部 | **只能由資料庫 trigger/RPC 寫**（前端不可直接寫） |
| `missions` / `tasks`（任務） | 登入者皆可讀 | 限管理員 |
| `teams` / `batches`（小隊/梯次） | 登入者皆可讀 | 限管理員 |
| 寵物/成就等 | 登入者皆可讀 | 本人或系統 trigger |

重點：**分數一律只能透過後端規則（trigger/RPC）變動**，前端永遠不能直接改 `profiles.score`。
這就堵掉「自己改分數」的漏洞。

---

## 5. 施工步驟（分階段，可逐步上線）

> 每階段都先在「測試用的 Supabase 專案」做，確認沒問題再上正式。

**階段 1：建帳號系統（不影響現有功能）**
- 開啟 Supabase Auth（選定的登入方式）
- `profiles` 加 `auth_user_id`、建 `user_roles` 表
- 寫「把現有學員對應到 auth 帳號」的遷移腳本

**階段 2：改前端登入/註冊**
- [app/page.tsx](../app/page.tsx) 的 `handleLogin` / `handleRegister` 改走 Supabase Auth
- 開機讀取 session 改用 auth（取代 `localStorage` 的 `nlp_mock_user_id`）

**階段 3：管理操作搬到後端**
- 審核、改分、刪除、管理小隊等改成 **Server Action（用 service role key）**，
  並在裡面驗證「呼叫者是不是管理員」
- 這樣前端就算被竄改也沒用，真正的權限在後端

**階段 4：收緊 RLS**
- 把第 4 節的政策套上去
- **逐表測試**每個功能還能跑（見第 7 節測試清單）

**階段 5：正式切換**
- 公告維護時間 → 套用 → 全面測試 → 開放

## 6. 風險與回滾

| 風險 | 對策 |
|---|---|
| 改壞導致學員登不進來/打不了卡 | 全程在**測試專案**做，正式切換選課程空檔 + 先備份 |
| 現有學員對應不到 auth 帳號 | 遷移腳本先跑「試跑」對帳，確認每人都對到才正式 |
| RLS 設太嚴卡到正常功能 | 逐表、逐功能測試清單（第 7 節），有問題即時放寬 |
| 想退回舊版 | 切換前用 `npm run backup` 備份；保留舊分支可一鍵還原 |

---

## 7. 測試清單（切換前每項都要過）

- [ ] 新學員可註冊、可登入
- [ ] 舊學員用手機可登入、看到自己的資料
- [ ] 學員可打卡、可上傳證明、可看排行榜
- [ ] 學員**不能**改自己分數（用開發者工具試打 API 應被擋）
- [ ] 學員**不能**看到/改別人的私密資料
- [ ] 隊長可審核自己小隊、看小隊成員
- [ ] 管理員可改分、管理任務/小隊/梯次、刪見證照片
- [ ] 見證牆、寵物進化等既有功能正常

---

## 8. 時間估算

- **開發 + 測試：約 1～2 週**（視登入方式與測試深度）
- 不含等待你決定登入方式的時間
- **強烈建議排在兩期課程之間**，不要在課程進行中做

## 9. 現在（v1 期間）零風險的緩解措施

在 v2 完成前，這些不用改程式就能降低風險：
- 別把網址公開貼到不特定大眾，只發給學員
- 定期 `npm run backup`（已建工具）→ 被亂改可還原
- 偶爾看排行榜，分數異常可從 `score_logs` 查異動
