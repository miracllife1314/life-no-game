# TESTING — 測試策略

> 「完成的定義」（`docs/AI_RULES.md` 第 10 節）要求測試通過才能宣稱完成，本檔定義怎麼測。
> 原則：測試量力而為，但**金流、權限、交易三類沒有豁免**。

## 0. AI 驗證通道（必備基建，對任何新系統通用性最高的一條）

**每個系統必須提供 AI 可用的端到端驗證通道**，擇一即可：

1. **測試帳號＋種子資料**（local / staging，帳號涵蓋每種主要角色）。
2. **免密身分切換**（僅限開發環境，可一鍵以任一角色視角操作）。
3. **staging 自動登入**（環境變數注入測試身分）。

為什麼是必備而非建議：沒有通道，`docs/AI_RULES.md` §10 第 4 點「實際執行過受影響的流程」
**物理上做不到**——AI 永遠只能報「已部署，待驗證」，本該 AI 自己 30 秒跑一次就抓到的 bug
會一路漏到使用者面前（實戰教訓：AI 登不進主要使用者頁面 → 播放器 bug 漏到使用者手上）。
**此基建的防 bug 效益高於任何審查制度。**

安全邊界：通道只存在於 local / staging，**不得通到 production**；
測試帳號不得含真實個資（見第 5 節測試資料原則）。
AI 測不了的流程，必須明講「我測不了 X，需要你測 Y」（§10 協定），不得含糊帶過。

## 1. 測試分級策略

| 層級 | 測什麼 | 什麼時候必須 |
|---|---|---|
| 單元測試 | 純函式、金額計算、狀態機、驗證邏輯 | 金流 / 點數計算邏輯必寫 |
| 整合測試 | API / Server Action + 資料庫行為（冪等、UNIQUE、RLS） | 交易與權限相關功能必寫 |
| E2E / 手動驗證 | 完整使用者流程（報名 → 付款 → 完成） | 每次改到關鍵路徑時執行 |

## 2. 必測清單（不可豁免）

以下三類**必須有自動化測試**，不能只靠手動點看看：

1. **金流**：金額計算、退款金額、優惠折抵、四捨五入邊界。
2. **權限**：一般使用者不能越權讀 / 寫（拿別人的 id 打 API 應被拒）。
3. **交易防呆**：重複提交只成立一筆（冪等）、超賣擋下、無效狀態被排除。

對應風險見 `docs/TRANSACTION_RULES.md` 與 `docs/INCIDENT_PREVENTION.md`——那 12 個歷史坑就是最好的測試案例來源。

> 機器提醒：`scripts/preflight-check.sh` 第 9 項會在「變更碰到金流/交易檔案、卻沒有任何測試變更」時黃字提醒（不擋 commit）。提醒出現時，請補測試或在回報中說明為何本次可豁免。

## 3. 關鍵路徑回歸

每次部署前（或改到相關程式時），至少驗證：

```text
[ ] 登入 / 登出
[ ] 核心商業流程走通（<main_workflow>）
[ ] 結帳 / 報名成功一筆
[ ] webhook 收單處理正常（可用測試事件）
[ ] 後台能正常讀取列表與儲存
```

## 4. 手動驗證步驟格式

自動化測不到的部分，交付時用此格式列出（AI 回報時必附）：

```text
驗證項目：
前置條件：
操作步驟：1. … 2. … 3. …
預期結果：
實際結果：（AI 實際跑過才能填）
```

## 5. 測試資料原則

1. **不得用 production 資料測試**；不得把測試資料灌入正式資料庫（見 `docs/DEPLOYMENT.md`）。
2. 測試資料要能一眼識別（例：email 用 `test+xxx@example.com`）。
3. 測後清理，或使用可拋棄的 local / staging 環境。

## 6. Bug 修復規則

修一個 bug，**必附一個能重現它的測試**（或至少手動重現步驟），確認修復後測試轉綠。
沒有重現步驟的「應該修好了」不算修好——並將教訓記入 `docs/INCIDENT_PREVENTION.md`。
