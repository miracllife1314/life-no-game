## 3. 🔍 八大角色全維度診斷分析 (Multi-Role Analysis)

### 3.1 📋 Product Manager（產品經理視角）
* **核心關焦點**：產品需求完整度、品牌文案白牌化、學員跨期留存流程。
* **診斷結論**：
  1. **品牌文案白牌化**：目前 `lib/brand.ts` 運作良好，建議未來所有新功能（如生命數字、課程報名）的文字皆抽離至品牌設定檔，避免硬編碼。
  2. **跨期結業引導**：當學員 status 變為 `ended` 時，介面宜提供更溫暖的「本期圓滿結業 Banner」，並引導報名下一期。

---

### 3.2 🎨 UI/UX Commercial Product Designer（UI/UX 商用產品設計師視角）
* **核心關焦點**：商用視覺質感、Mobile-first 防破版、點擊觸控標籤、動畫順暢度。
* **診斷結論**：
  1. **手機版防破版與觸控面積**：頂部 Header 與微型按鈕需嚴格保持最小觸控面積（**≥ 44px × 44px**），防範小螢幕手機（如 iPhone SE）誤觸。
  2. **CSS 3D 渲染與顯卡硬體加速**：每日抽卡與神獸動畫建議加上 `will-change: transform; transform: translateZ(0);`，確保低階手機滑動時零卡頓。

---

### 3.3 🏗️ System Architect（系統架構師視角）
* **核心關焦點**：模組解耦、狀態管理架構、系統擴充性。
* **診斷結論**：
  1. **單體主頁面 (`app/page.tsx`) 模組化**：`app/page.tsx` 掌管了 SPA 的全域遊戲狀態。隨著功能成長，建議未來採用 React Context（如 `GameStateContext`, `UserAuthContext`）將資料流分層管理，避免全頁面過度重新渲染 (Re-render)。

---

### 3.4 💻 Developer（開發工程師視角）
* **核心關焦點**：代碼維護性、N+1 查詢風險、型別嚴密性 (`no-explicit-any`)。
* **診斷結論**：
  1. **避免 N+1 查詢**：在 `services/queries.ts` 與 `LeaderboardTab.tsx` 中，嚴格使用 Supabase Nested Select 批次一次性拉取關聯欄位（如 `submissions(*, tasks(name))`），嚴禁在迴圈與 `.map()` 內查詢 DB。
  2. **型別補完**：逐步清理 `services/queries.ts` 內的 `any` 宣告，改為 `types/index.ts` 的強型別介面，提升重構安全感。

---

### 3.5 🗄️ Database Engineer（資料庫工程師視角）
* **核心關焦點**：Schema 設計、高頻查詢索引 (Index)、RLS 存取控管。
* **診斷結論**：
  1. **建立 DB 高頻欄位索引**：建議在 Supabase SQL Editor 執行以下 Index 建置，大幅提升萬筆打卡與排行榜查詢效能：
     ```sql
     -- 高頻查詢欄位建立 Index
     CREATE INDEX IF NOT EXISTS idx_profiles_phone ON profiles(phone);
     CREATE INDEX IF NOT EXISTS idx_profiles_batch_id ON profiles(batch_id);
     CREATE INDEX IF NOT EXISTS idx_submissions_student_mission ON submissions(student_id, mission_id);
     CREATE INDEX IF NOT EXISTS idx_score_logs_student_id ON score_logs(student_id);
     ```

---

### 3.6 🛡️ Security Engineer（資安工程師視角）
* **核心關焦點**：個資防護、Service Role 金鑰隱私、防暴力破譯。
* **診斷結論**：
  1. **Service Role 邊界**：所有具備 Service Role 提權權限的操作均收納於 `/api/...` 後端端點，前端 Client 僅能存取 `NEXT_PUBLIC_SUPABASE_ANON_KEY`。
  2. **個資隱私**：學員手機號碼 (`phone`) 在 RLS 政策中嚴格限制，非管理員不可查詢他人手機。

---

### 3.7 🧪 QA Engineer（測試工程師視角）

  2. **離線/慢速網路防護**：提交打卡證明與圖片上傳時，確保有 Loading 遮罩防護，防止學員因連點造成重複提交。

### 3.8 🚀 DevOps Engineer（部署與維運工程師視角）
* **核心關焦點**：Vercel 部署自動化、環境變數隔離、資料庫備份機制。
* **診斷結論**：
  1. **環境變數備援**：程式碼中已內建安全的預設備援機制，即使 Vercel 環境變數誤跑，系統亦會自動鎖定並連線至專屬揚升資料庫。
  2. **自動化備份**：定期執行 `npm run backup` 工具，將資料庫快照匯出為本地備份。
