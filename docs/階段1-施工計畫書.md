## 步驟 1：補資料庫索引

### 會改什麼
只新增索引，不改表結構/資料/程式。全部 `IF NOT EXISTS` + `CONCURRENTLY`（建立時不鎖表）。

**必建（主載入熱路徑）**
- `submissions(student_id)`、`score_logs(student_id)`、`user_pets(student_id)`、`user_achievements(student_id)`、`course_attendance(student_id)`、`student_notes(student_id)`
- `missions(batch_id)`

**應建（次熱）**
- `submissions(mission_id)`、`score_logs(submission_id)`
- `profiles(batch_id)`、`profiles(team_id)`、`profiles(profile_id)`、`profiles(phone)`
- `teams(batch_id)`、`missions(template_id)`、`batch_mission_templates(batch_id)`

`id` 為主鍵已自動索引，不建。既有 5 索引（login_attempts、auth_user_id、witness）會自動跳過。

### 影響功能
載入/排行榜/見證牆/審核/打卡判斷變快；寫入多極微小索引維護成本。**功能行為零改變。**

### 風險 / Rollback
極低（CONCURRENTLY 不鎖表、IF NOT EXISTS 不重複）。Rollback：`DROP INDEX IF EXISTS`。

### 驗收
`EXPLAIN ANALYZE` 前後（Seq Scan→Index Scan）、`security-check` 16/16、`test-flows` 14/14、手動點學員端。

### 預估改善
300~1000 人資料量時 `.in(student_id)` 類查詢快數倍～數十倍。「1000 人能不能用」的關鍵。

## 步驟 3：收緊 Storage 政策

### 會改什麼
現行 `using(bucket_id='proof-images') with check(bucket_id='proof-images')`（過寬，任何人可寫/刪）→ 改為「公開讀、寫入需登入身分、刪除限本人/管理」。proof-images / pet-images 分別設計。

### 影響功能
封掉「F12 刪改別人證明圖／匿名灌爆空間」。**需先實測打卡上傳是否帶 auth session，避免誤傷打卡。**

### 風險 / Rollback
中（三步最高，可能誤傷上傳）→ 測試庫先做、先測打卡上傳成功才動正式。Rollback：`DROP POLICY` 重建舊政策。

### 驗收
測試庫打卡上傳成功（最重要）、管理員上傳寵物圖成功、匿名寫/刪被拒、security-check/test-flows 通過。
