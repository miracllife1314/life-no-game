# 安全強化 施工計畫書
## 方案：保留「姓名+電話」登入 → 後端核發真實身分 → 開啟 RLS 鎖定

> 本文是「施工計畫」，**先不寫任何程式**。你看完、確認方向沒問題後，我才會按階段動工。
> 全程在 **測試庫** 做完、你實測通過，**正式站完全不動**；最後再另外規劃正式站遷移。

---

## 0. 一句話總結

把現在的「假登入（只比對姓名電話、全站匿名）」改成：**學員一樣只打姓名+電話，但背後由我們的伺服器幫他換發一張「真實的 Supabase 通行證(JWT)」**。有了真實身分，資料庫的 RLS 才有辦法判斷「你是誰、能看/改什麼」，才能擋掉「F12 灌分、代別人打卡」這類漏洞。

**學員端體驗完全不變**（不用收簡訊、不用記密碼），改的是背後的身分核發機制。

---

## 1. 為什麼非這樣不可（現況的問題）

| 現況 | 問題 |
|---|---|
| 登入只比對姓名+電話，不向 Supabase 要身分（[page.tsx:418,544](../app/page.tsx#L418)） | 全站每個人都是同一個「匿名(anon)」身分 |
| RLS 全開（`using(true)`） | 任何人開 F12 都能讀/改任何資料 |
| 後台動作（改分、審核、改帳號）也都在前端用匿名身分做 | 沒有任何後端把關 |

**核心觀念**：RLS（資料庫的權限門禁）要能運作，前提是「每個人都有真實身分」。所以「換真實登入」和「開 RLS」必須一起做，不能只做一半——只開 RLS 不換登入，會因為 `auth.uid()` 是空的，導致**全站當場全黑**。

## 2. 整體架構（改完後長這樣）

```
學員輸入 姓名+電話
        │
        ▼
[我們的後端 /api/auth/login]  ← 用 service role 金鑰（只在伺服器，前端碰不到）
   1. 查 profiles 找到這個人
   2. 確保他在 Supabase Auth 有一個帳號（用手機當人別 key）
   3. 把他「所有期別」的 profiles 綁上同一個 auth 身分
   4. 產生一張一次性通行票(token)，回傳給前端（過程不寄任何簡訊/email）
        │
        ▼
[前端] 用這張票 verifyOtp() → 拿到真實 Session(JWT)
        │
        ▼
[之後所有查詢自動帶上 JWT]
        │
        ▼
[Supabase 資料庫]
   RLS 門禁用 auth.uid() 判斷：
     · 學員：只能改自己的個資、只能幫自己打卡、只能看自己的分數明細
     · 隊長：可審核自己隊員的打卡
     · 大隊長：全部可管（is_admin() 放行）
   防改分 Trigger：非管理員改 score 一律擋下
```

**省工關鍵**：大隊長、隊長也走同一套登入拿到真實身分；RLS 政策用 `is_admin()` / `is_captain_of()` 放行他們，所以**現有後台前端程式幾乎不用重寫**，只是從「匿名能做」變成「具備管理身分才能做」。

## 4. 施工階段（全部在測試庫，逐階段可獨立驗證、隨時可回復）

### 階段 0：後端核發身分（**對學員無感，先不開 RLS**）
目標：讓每個人登入後「除了照舊能用，還多拿到一張真實 JWT」。此時 RLS 還沒開，App 行為完全不變，純粹先把身分基礎建起來。

1. 資料庫：`profiles` 加欄位 `auth_user_id uuid UNIQUE`（對應規格書第 1 步）。
2. 新增後端路由 `app/api/auth/login/route.ts`（目前 `app/api` 是空的，全新建立）：
   - 收 `{ name, phone }`。
   - service role 查 `profiles`（姓名+電話）。查無 → 回「查無此帳號」。
   - 以「純數字手機」合成內部 email：`{phone}@nlp.local`（僅內部識別，不會真的寄信）。
   - `auth.admin` 確保該 email 的 Auth 使用者存在（沒有就建立、`email_confirm: true`）。
   - 把這個人**所有期別**的 `profiles` 列都綁上同一個 `auth_user_id`（多期共用同一身分）。
   - `auth.admin.generateLink({ type: 'magiclink', email })` 取得 `token_hash`，回傳給前端。
3. 前端 `handleLogin` 改為：先打 `/api/auth/login` 拿 `token_hash` → `supabase.auth.verifyOtp({ token_hash, type: 'email' })` 建立 Session → 之後沿用現有流程載入資料。
4. 保留舊的「假登入」程式在一個開關後面（feature flag），萬一新流程出問題可立即切回。

**驗收**：各角色登入後，瀏覽器 console 打 `await supabase.auth.getUser()` 應拿到真實 user；`profiles.auth_user_id` 已正確綁定；App 一切照舊。

---

### 階段 1：權限政策就位（**寫好但先休眠，RLS 仍關閉**）
把門禁規則都裝好，但先不通電，確保規則本身不會打架。

1. 建輔助函式（`SECURITY DEFINER`，避免 RLS 遞迴）：
   - `public.is_admin()`：用 `auth.uid()` 查自己的 profile，`role = 'admin'` 回 true。
   - `public.is_captain_of(student_id)`：caller 是該學員所屬小隊的隊長則 true。
2. **改寫防改分 Trigger**（比規格書更嚴謹）：
   - 原規格只擋 `anon`；但改造後大家都是 `authenticated`，光擋 anon 不夠。
   - 改成：`profiles.score` 有變動且 **`NOT is_admin()`** → 直接 `RAISE EXCEPTION`。這樣**不論匿名或一般學員**都改不動分數，只有管理員與系統 trigger 能改。
3. 寫好 RLS 政策（先 `CREATE POLICY`，但 **不** `ENABLE ROW LEVEL SECURITY`）：
   - `profiles`：SELECT 全部可讀（排行榜/組員名單需要）；UPDATE 限本人或 `is_admin()`；INSERT 限 `is_admin()`（報名改走後端 service role）。
   - `submissions`：SELECT 可讀（見證牆/進度需要）；INSERT 限「student_id 對應到 `auth.uid()`」（擋代打卡）；UPDATE/DELETE 限 `is_admin()` 或 `is_captain_of(student_id)`。
   - `score_logs`：SELECT 限本人或 `is_admin()`/隊長；**前端完全不能 INSERT/UPDATE/DELETE**（由 `SECURITY DEFINER` trigger 自動寫，已確認會繞過 RLS）。
   - 其餘參考表（missions/tasks/pets…）維持 RLS 關閉、照常可讀。

**驗收**：SQL 全部能順利執行、無語法/相依錯誤；此時 RLS 仍關，App 照舊。

---

### 階段 2：通電鎖定 + 攻擊測試（測試庫）
1. 對 `profiles` / `submissions` / `score_logs`（+ `batch_mission_templates`）`ENABLE ROW LEVEL SECURITY`。
2. **全角色回歸測試**（逐項勾）：
   - 學員：登入、每日打卡、上見證牆、看排行榜、看自己分數明細、改自己個資。
   - 隊長：審核/退回自己隊員打卡、看隊員進度。
   - 大隊長：改分、改帳號、發任務、刪資料、檢視學員(代打卡) 全可用。
3. **F12 攻擊測試**（規格書第三部分）：
   - 嘗試 `update profiles set score=9999` → 應被防改分 trigger 擋下。
   - 嘗試用別人 student_id `insert submissions` → 應被 RLS 擋下。
   - 嘗試直接 `insert score_logs` → 應被擋下。

**驗收**：上面正常功能全綠、三項攻擊全被擋。

## 5. 風險與「一鍵回復」

| 風險 | 回復方式 |
|---|---|
| 開 RLS 後有功能讀不到/寫不進 | `ALTER TABLE xxx DISABLE ROW LEVEL SECURITY;`（逐表）即可**立刻恢復全開**，因為資料表仍授權 anon/authenticated 讀寫。 |
| 新登入流程有 bug 進不來 | 前端 feature flag 切回舊「假登入」。 |
| Schema 改壞 | 已先備份，`npm run restore` 還原。 |
| 正式站出狀況 | 上面三招都可用；且階段 3 才碰正式站，前面都在測試庫驗證過。 |

**整套設計原則：每一步都可獨立回復，正式站永遠是最後、且可秒退。**
