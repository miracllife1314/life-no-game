# OBSERVABILITY — 監控、日誌、備份與還原

> 涉及部署 / production / 事故時，搭配 `docs/DEPLOYMENT.md`。
> 商用系統出事時，這份文件決定你能不能快速止血與復原。

## 1. 錯誤監控

1. 建議接入錯誤監控服務（例如 **Sentry**），前後端皆納入。
2. 未捕捉的例外、API 5xx、前端 crash 都要進監控。
3. 監控要能標記環境（production / staging）與版本（release / commit）。
4. 監控事件不得包含個資與機密（見第 3 節）。

## 2. log 分級策略

| 等級 | 用途 | 範例 |
|---|---|---|
| `debug` | 開發除錯，正式環境預設關閉 | 變數內容、流程節點 |
| `info` | 正常業務事件 | 訂單建立、使用者登入 |
| `warn` | 可容忍但需注意 | 重試、降級、逾時 |
| `error` | 需要處理的錯誤 | 交易失敗、外部服務錯誤 |

規則：

1. 正式環境不輸出 `debug`。
2. 每筆 error log 要能定位（request id / user id 匿名化 / 時間 / 模組）。

## 3. 不記錄個資與敏感資訊原則

1. **禁止** log 密碼、token、service_role key、金流卡號、完整個資。
2. 需要關聯使用者時，用匿名化 id，不記可識別個資。
3. 不得 `console.log` 敏感資料（見 `docs/SECURITY_CHECKLIST.md` 第 13 項）。

## 4. 告警門檻

1. error 率超過門檻（例如 5 分鐘內 error > N）觸發告警。
2. 關鍵流程（結帳 / 付款 / webhook）失敗即時告警。
3. 資料庫連線 / 額度 / 儲存空間接近上限時告警。

## 5. 資料庫備份頻率

1. production 至少每日自動備份（Supabase 內建備份 + 視需求額外匯出）。
2. 重大變更（migration 上正式）前，先手動備份一次。
3. 備份保留週期需明確定義（例如保留 N 天 / N 週）。

## 6. 還原演練流程

1. 定期（例如每季）做一次還原演練，確認備份真的可用。
2. 演練在 staging 進行，不得直接動 production。
3. 記錄還原耗時（RTO）與可接受資料遺失量（RPO）。

## 7. 誤刪資料救援步驟

```text
1. 立即停止相關寫入，避免覆蓋更多資料。
2. 判斷影響範圍（哪些表、哪些列、時間點）。
3. 從最近可用備份 / PITR（Point-in-Time Recovery）還原到 staging 驗證。
4. 比對差異，僅回填受影響資料，避免整庫覆蓋。
5. 確認無誤後，於維護窗口套用到 production。
6. 事後補記錄到第 8、9 節。
```

## 8. production 事故紀錄格式

```text
事故時間：
發現方式：
影響範圍：
嚴重程度：
處理過程：
止血時間：
根本原因：
```

## 9. 事件後檢討格式（Postmortem）

```text
事故摘要：
時間軸：
根本原因：
做得好的地方：
做得不好的地方：
後續行動項（負責人 / 期限）：
需新增的防範規則（同步到 docs/INCIDENT_PREVENTION.md）：
```
