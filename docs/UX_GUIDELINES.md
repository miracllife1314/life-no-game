# UX GUIDELINES — UX、轉化率與後台編輯器規範

> 涉及 UI / 前台流程 / 後台編輯器時，先讀本檔與 `docs/BRAND_UI_SYSTEM.md`。
> 修改前台轉化流程或後台編輯器時，依 `docs/AI_RULES.md` 第 8 節同步更新本檔。

## 一、前台轉化率

1. **降低阻力**：報名 / 結帳步驟越少越好，非必要資訊延後收集。
2. **不第一步強迫註冊**：若商業情境允許，訪客可未登入先填必要資料並送出；帳號建立、推薦碼歸戶、付款憑證可延後處理。
3. **不建議**用手機末四碼、生日、預設密碼當正式密碼。
4. **建議**用 OTP、Email magic link、付款完成後帳號啟用連結、首次登入設定密碼。
5. **延後收集非必要資料**：表單欄位不可過多。
6. **白話錯誤訊息**：不要丟技術錯誤碼給使用者。
7. **使用者知道下一步**：每個狀態都告訴使用者接下來要做什麼。

## 二、後台複雜編輯器

1. **draft state**：大量欄位編輯時，改動先進本地草稿狀態，不即時打 API。
2. **不逐格打 API**：不得每改一個下拉 / 欄位就立即送出請求。
3. **Sticky Save Bar**：底部固定儲存列，提示「有未儲存的變更」。
4. **批次送出**：儲存時把變更批次送出（PATCH API 或 RPC）。
5. **未儲存提醒**：離頁前若有未儲存內容，要攔截提醒。
6. **儲存狀態完整**：loading / success / error / partial failure 都要處理。
7. **高風險二次確認 + audit log**：高風險單筆操作要二次確認、立即寫入並記 audit log。
8. **搜尋 / 篩選 / 分頁**：後台表格必須支援搜尋、篩選、分頁或 lazy loading（見 `docs/AI_RULES.md` 效能相關與 anti-n-plus-one skill）。

## 三、表單與錯誤提示

1. **必填欄位清楚**：一眼看出哪些必填。
2. **錯誤訊息靠近欄位**：就近顯示，不要只在頂部。
3. **錯誤訊息告訴下一步**：說明怎麼修正，而非只說「錯誤」。
4. **成功狀態明確**：送出成功要有明確回饋。
5. **loading / disabled 狀態完整**：送出中禁止重複提交（但真正防重複在後端，見 `docs/TRANSACTION_RULES.md`）。

## 四、符合人性的簡單操作（硬性檢查）

UI/UX 設計必須優先確認以下 13 項；**簡單、直覺、可信任的優先順序高於炫技動畫與複雜視覺**：

1. 使用者是否知道目前在哪裡。
2. 使用者是否知道下一步要做什麼。
3. 每個畫面是否有清楚的主要目標。
4. 操作步驟是否可以再減少。
5. 系統能自動取得的資料，不要叫使用者手動尋找或重填。
6. 新手是否看得懂文字與術語（避免工程用語）。
7. 錯誤訊息是否清楚說明原因與處理方式（見第三節）。
8. 成功後是否清楚告知結果與下一步。
9. 是否有過多按鈕、選項或資訊造成選擇困難。
10. 常用功能是否容易找到。
11. 危險操作是否有適當確認（見第二節第 7 點）。
12. 表單是否只要求真正需要的資訊（見第一節第 5 點）。
13. 使用者中途離開後，是否容易知道如何繼續。
