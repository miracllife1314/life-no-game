# INCIDENT PREVENTION — 歷史重大 Bug 防範

> 通用預設防範庫。每次踩到新坑，依同一格式新增一筆。
> 涉及金流 / 交易時，搭配 `docs/TRANSACTION_RULES.md`。

**紀錄格式：**

```text
問題名稱：
發生原因：
影響範圍：
錯誤做法：
正確做法：
未來防範規則：
測試案例：
```

---

## 1. 重複報名冪等死鍵

- **發生原因**：使用者連點或網路重送，同一報名被寫入多筆。
- **影響範圍**：名額被灌爆、資料重複、對帳困難。
- **錯誤做法**：只在前端 disabled 按鈕。
- **正確做法**：冪等鍵 + UNIQUE(user_id, event_id)，重複寫入被 DB 擋下。
- **未來防範規則**：任何「一人一次」的動作，DB 層必須有 UNIQUE。
- **測試案例**：同一請求連送 5 次，只成立 1 筆。

## 2. 退款點數重複返還

- **發生原因**：退款流程無狀態機，重試造成重複返還。
- **影響範圍**：點數 / 金額被灌，財務損失。
- **錯誤做法**：直接對餘額 `+= amount`，無冪等。
- **正確做法**：退款狀態機 + 冪等鍵 + ledger 記錄每筆異動。
- **未來防範規則**：金額 / 點數異動一律走 ledger，禁止裸改餘額。
- **測試案例**：對同一退款重試 3 次，只返還 1 次。

## 3. 高流量結帳鎖表

- **發生原因**：結帳交易鎖範圍過大或鎖整表，高併發時卡死。
- **影響範圍**：結帳逾時、使用者流失。
- **錯誤做法**：交易內做大量非必要查詢、鎖粒度過粗。
- **正確做法**：縮小交易範圍、只對關鍵行 FOR UPDATE、用條件式 UPDATE 扣庫存。
- **未來防範規則**：交易內不做慢查詢；鎖到行不鎖到表。
- **測試案例**：模擬 100 併發結帳，無死鎖、無超賣。

## 4. 優惠碼重複核銷

- **發生原因**：核銷檢查在應用層做，併發下重複通過。
- **影響範圍**：一碼多用，行銷成本失控。
- **錯誤做法**：先 SELECT 判斷再 INSERT，中間有競態。
- **正確做法**：UNIQUE(coupon_id, user_id) 或使用記錄唯一約束，靠 DB 擋。
- **未來防範規則**：核銷唯一性放 DB 層。
- **測試案例**：同碼同人併發核銷，只成立 1 次。

## 5. Webhook 重複處理

- **發生原因**：金流商 webhook 會重送，未去重。
- **影響範圍**：重複入帳、重複發貨。
- **錯誤做法**：每次 webhook 都直接處理。
- **正確做法**：以事件 id 建立處理記錄（UNIQUE），已處理則直接回 200 忽略。
- **未來防範規則**：所有 webhook 先查去重表再處理。
- **測試案例**：同一事件重送 3 次，只入帳 1 次。

## 6. cancelled / expired / failed 狀態未排除

- **發生原因**：統計或扣減時把無效狀態也算進去。
- **影響範圍**：名額 / 金額計算錯誤。
- **錯誤做法**：`select count(*)` 不加狀態過濾。
- **正確做法**：查詢明確 `where status not in ('cancelled','expired','failed')`。
- **未來防範規則**：所有計數 / 扣減都要顯式過濾狀態。
- **測試案例**：混入無效狀態資料，計算結果不受影響。

## 7. admin 權限只靠前端判斷

- **發生原因**：權限只在前端隱藏按鈕，API / RLS 無保護。
- **影響範圍**：一般使用者可直接打 API 越權。
- **錯誤做法**：`if (isAdmin) show()`，後端不驗。
- **正確做法**：權限在 RLS / server 層強制，前端隱藏只是體驗。
- **未來防範規則**：權限判斷以 DB / server 為準。
- **測試案例**：以一般使用者身分直接打 admin API，被拒。

## 8. localStorage 模擬登入混入正式流程

- **發生原因**：原型階段用 localStorage 假登入，未清理就上線。
- **影響範圍**：身分可偽造，安全破口。
- **錯誤做法**：用 localStorage 標記登入狀態當正式判斷。
- **正確做法**：正式登入走 Supabase Auth session，禁止假身分。
- **未來防範規則**：Prototype 假資料不得混入正式功能（`docs/AI_RULES.md` 第 3 節）。
- **測試案例**：竄改 localStorage 無法取得受保護資源。

## 9. RLS using(true) / check(true)

- **發生原因**：為了快速通過，policy 寫成永遠成立。
- **影響範圍**：等同關閉 RLS，全表可讀寫。
- **錯誤做法**：`using (true)` / `with check (true)`。
- **正確做法**：以 `auth.uid()` 與角色條件限制。
- **未來防範規則**：preflight 掃描擋下 using/check(true)。
- **測試案例**：他人資料無法被越權存取。

## 10. N+1 查詢造成後台卡死

- **發生原因**：列表在 loop 內逐筆查關聯資料。
- **影響範圍**：後台列表 / 儀表板嚴重變慢。
- **錯誤做法**：`for (row) { db.query(row.id) }`。
- **正確做法**：收集 IDs → `.in()` 批量查 → 記憶體用 Map 組裝（見 anti-n-plus-one skill）。
- **未來防範規則**：禁止在 loop 中查 DB / 打 API；列表必分頁。
- **測試案例**：1000 筆列表查詢次數為常數級，非線性。

## 11. Zsh 動態路由路徑未加雙引號

- **發生原因**：路徑含 `[id]` 等特殊字元，zsh glob 展開失敗。
- **影響範圍**：指令找不到檔、誤刪、腳本中斷。
- **錯誤做法**：`rm app/[id]/page.tsx`（未加引號）。
- **正確做法**：路徑一律加雙引號 `"app/[id]/page.tsx"`。
- **未來防範規則**：所有含特殊字元路徑都加雙引號。
- **測試案例**：對含 `[]` 路徑執行指令正常。

## 12. 文件未同步更新導致 AI 依照過期文件開發

- **發生原因**：改了程式 / DB，但 docs 沒同步，AI 讀到舊資訊。
- **影響範圍**：AI 依過期規範誤判、決策錯誤。
- **錯誤做法**：改完程式不更新 docs，留到以後。
- **正確做法**：依 `docs/AI_RULES.md` 第 8 節「文件同步熔斷」，同一次任務同步更新。
- **未來防範規則**：每次任務結束檢查 docs 是否過期並更新 `logs/ai_sync_log.md`。
- **測試案例**：抽查任一 docs 與現況一致。

## 13. 規範文件雙頭馬車（兩套規範平行維護）

- **發生原因**：匯入模板或既有專案本就有規範，導致同一主題有兩份文件（例如兩份 AI_RULES、兩套 DB 規範）並存。
- **影響範圍**：兩份日久分岔，AI 讀到過期那份 → 依「不存在的現況」誤判；常在 brownfield 匯入後發作。
- **錯誤做法**：兩套規範都留著、都偶爾改一下，沒指定誰是權威。
- **正確做法**：指定一份為單一事實來源（`docs/AI_RULES.md`），其餘檔頭標「歷史參考，現行見 docs/AI_RULES.md」並停止平行維護（見 `docs/AI_RULES.md` 第 8 節「單一事實來源」）。
- **未來防範規則**：發現同主題多份規範時，AI 必須主動提報並建議整併，不得默默沿用其一。
- **測試案例**：全庫搜尋同名 / 同主題規範文件，確認每主題只有一份權威。
