本文件記錄「目前不急著實作，但未來當專案進入更高商用複雜度時，才回流母版的能力」。

母版 v1.4.0 已定版，**不因假設性需求提前重構**。以下項目只有在**真實專案出現明確需求**時，才進入母版升版。

> 與其他「未來相關」文件的界線：
> - **本清單（FUTURE_BACKLOG）** = 已知、**刻意延後**、由設計者主動判斷的母版能力。
> - **`logs/template_feedback.md`** = 實戰中**新發現**的問題（察覺→收集→回流，見 `docs/FEEDBACK_LOOP.md`）。
> - 三項待辦將來動手時，**全部屬 `docs/AI_RULES.md` §5 高風險**（schema / RLS / 金流），須走停手提方案流程，不得當一般任務隨手做。

## 1. 多租戶隔離規範 tenant_id / org_id

> ⚠️ **這一項與其他兩項不同：建置可以晚，架構賭注必須早下。**
> 把 `org_id` 回填進一個已有數十張表的系統是地獄級遷移。因此**每開一個新專案，在 kickoff 就要決定「這系統這輩子會不會變多租戶？」**——若有可能，即使先不建組織級 RLS，資料表設計也應**預留 `org_id` 欄位、避免做出日後擋路的設計**。這個決策不能延後，只有「建置」可以延後。

### 觸發條件（開始「建置」的時機）

當系統開始支援：多個公司 / 多個團隊 / 多個講師 / 多個組織共用同一套系統 / 不同組織之間資料必須完全隔離。

### 未來要補的內容

* tenant_id / org_id 資料庫欄位規範
* 組織級 RLS 設計（在 user 級之上再加一層）
* 使用者與組織關聯表（一人可屬多組織）
* admin / owner / member 組織角色權限
* 防止 A 組織讀到 B 組織資料的檢查清單

### 目前狀態

暫不建置。母版先維持個人級與角色級權限規範。**但新專案 kickoff 時仍須做「是否預留 org_id」的架構決策，並記入該專案 `logs/decision_log.md`。**

## 2. 複式簿記 Double-Entry Bookkeeping 規範

### 觸發條件

當系統開始涉及：分潤 / 佣金 / 代理商獎金 / 多方拆帳 / 退費扣回 / 報稅對帳 / 財務稽核。

### 未來要補的內容

* 借貸雙邊帳務模型
* account / journal_entries / journal_lines 設計
* 每筆金流異動需借貸平衡
* 退款、扣回、折讓、佣金沖銷規則
* 財務稽核與對帳檢查清單

### 目前狀態

暫不實作。一般點數、優惠券、票券、訂單系統，可先使用 ledger / audit log / transaction log 模式（見 `docs/TRANSACTION_RULES.md`），不提前套用完整會計模型。

## 3. GitHub Actions CI/CD 自動檢查範本

> 三者中**最輕量**，但仍遵守「有真實需求才回流」——不因為它輕就提前建。

### 觸發條件

當專案開始：頻繁部署 / 多 AI 協作 / 多分支開發 / 接近正式上線 / 需要 push 後於雲端自動檢查（本機 hook 不足以涵蓋團隊協作時）。

### 未來要補的內容

* GitHub Actions workflow 範本
* npm run build / npm run lint / TypeScript check
* preflight-check（RLS 紅線、service_role 洩漏、migration/RLS 基礎檢查）

### 目前狀態

暫不實作。母版現階段靠本機雙層 hook（pre-commit 快檢 + pre-push build）已足夠單人開發；CI 是「團隊協作 / 多分支」出現後的升級，非現在。

## 4. preflight「檔案行數」軟提醒（R7 候選，使用者已裁決暫不做）

> 來源：學院一週實戰回饋 R7（2026-07-16）——CoursePlayer.tsx 長到 ~600 行，
> 違反 Karpathy #5（AI_RULES §9 第 5 點）但無任何提醒。
> 母版主裁決（2026-07-16）：**暫不做**——誤報風險中等（大檔只改一行也會被唸），
> 黃字噪音疲勞會侵蝕其他提醒的權威（正是 🛡️ 全域宣告 0% 執行率的成因類型）。

### 觸發條件

「大檔失控」在後續專案**重複發生**（同類回饋 ≥2 次）時，再升級為 preflight 黃字：
本次變更觸及的程式檔 >400 行即提醒考慮拆分（僅提醒不擋）。

### 目前狀態

暫不實作。維持 AI_RULES §9 Karpathy #5 文字規則（超過 300 行主動模組化拆分）。

## 使用原則

1. 不因假設性商業需求提前實作。
2. 只有當真實專案遇到明確需求，才回流母版。
3. 母版保持輕量、可執行、可實戰。
4. 小改要快，中改要穩，大改要嚴。
5. 不為了追求完整而犧牲一人公司的開發流暢度。
6. 例外——多租戶的「架構決策」不受本原則延後（見第 1 項），只有「建置」可延後。

---

## 目前結論

母版 v1.4.0 維持定版。下一步重點**不是繼續擴充母版，而是投入實戰專案**，透過真實開發回饋再決定是否升版。
