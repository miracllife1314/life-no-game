# 維護性重構 計畫書（page.tsx 瘦身）

> 目標：把 3125 行的巨型 `app/page.tsx` 拆成「小而清楚」的模組，**完全不改任何功能行為**（學員/後台操作一模一樣），只是讓程式更好維護、以後加功能更快更安全。
> 原則：**小步、漸進、每步驗證、全程測試庫**。正式站完全不動；要上線時再照規矩問你。

---

## 一、現況（為什麼維護性只有 5.5）

| 指標 | 數字 |
|---|---|
| `page.tsx` 行數 | **3125** |
| `useState` | 51 個 |
| `handleXXX` 業務函式 | 57 個 |
| 資料拼接/輔助函式 | ~30 個 |
| 渲染 JSX（各分頁） | 數百行 |

問題：**所有狀態 + 所有邏輯 + 所有畫面**都在一個檔案，牽一髮動全身——這次安全升級會那麼累、那麼容易踩雷，一半原因就是這個。

---

## 二、重構原則（安全第一）

1. **零行為改變**：只搬家、不改邏輯。搬完功能要跟現在 100% 一樣。
2. **一次一小塊**：每抽出一塊，立刻 `npm run build` + 本機點一輪 + `npm run security-check`，綠了才繼續。
3. **全程測試庫**：只在本機/測試庫做。
4. **每階段獨立可回退**：用 git 一階段一個 commit，出事可退回上一步。
5. **先抽風險最低的，再抽核心的**：由外而內。

---

## 三、分階段計畫（由低風險到高槓桿）

### 📌 階段 1：抽出「純工具函式」（風險：極低）
把不依賴畫面狀態的純函式搬到 `lib/`：
- `getMondayOfWeek`、`getChineseNumber`、`roleLabel`、`removeStorageImageByUrl`（→ `lib/helpers.ts`）
- 既有的資料拼接邏輯整理（`joinedSubs / joinedUserPets / joinedCandidates …` 共 ~10 個）→ 抽成 `useJoinedData(rawLists)` 一個 hook（`hooks/useJoinedData.ts`），回傳拼好的資料。
- **預估**：page.tsx 減 ~250 行。**幾乎零風險**（純資料轉換）。

### 📌 階段 2：抽出「UI 回饋」hook（風險：低）
- `showToast`、`triggerConfetti`、`triggerScoreFloat`、toast/confetti 狀態 → `hooks/useUiFeedback.ts`。
- page.tsx 改成 `const { showToast, triggerConfetti } = useUiFeedback()`。
- **預估**：減 ~120 行。

### 📌 階段 3：把 57 個 handler 依「領域」抽成 hook（風險：中，**最大槓桿**）
這是重點。把 handler 依功能分群，每群一個 hook（放 `hooks/`）：

| Hook | 包含的 handler |
|---|---|
| `useAuth` | login / register / logout / switchCohort / initSession |
| `useStudentActions` | checkIn / registerCourse / toggleCell |
| `usePetActions` | evolvePet / selectEvolutionLine / levelUpPet / awardPetSkin |
| `useReview` | reviewSubmission / markAttendance / saveNote |
| `useWitness` | hideWitness / deleteWitness |
| `useAdminPeople` | profiles / teams / squadRoles / captainCandidates / quickAssignCaptain |
| `useAdminContent` | batches / missionTemplates / generateMissions / tasks / announcements / courses / achievements |
| `useAdminMisc` | manualAdjustScore / pets / cards / decks |

做法：每個 hook 接收它需要的「狀態 + setter + supabase + showToast + fetchData」當參數，回傳該群 handler。page.tsx 變成呼叫這些 hook、把 handler 接到各分頁。
- **預估**：把 ~1800 行邏輯移出 page.tsx。**每抽一個 hook 就驗一次**（共 8 次小驗收）。

### 📌 階段 4（選做，較大）：導入輕量狀態容器，消除「參數一大串」
階段 3 後，hook 還是要傳一堆 state/setter。最後可導入 **Zustand**（或 React Context）當共用狀態庫，讓 hook 直接取狀態、不用一個個傳。
- 好處：擴充性大幅提升（6.5 → 8+），加新功能不用再動 page.tsx。
- 風險中等，**建議階段 1～3 穩定後再評估要不要做**。

---

## 四、預期效果

| | 重構前 | 重構後（階段 1～3） |
|---|---|---|
| page.tsx 行數 | 3125 | 約 **600～800**（只剩「組裝」與 JSX） |
| 邏輯位置 | 全在一檔 | 依領域分散在 `hooks/`、`lib/` |
| 加新功能 | 動巨型檔、易踩雷 | 動對應的小 hook，影響面小 |
| 維護性評分 | 5.5 | 預估 **8+** |
| 擴充性評分 | 6.5 | 預估 **7.5+**（做了階段 4 更高） |

---

## 五、風險與回退

- **最大風險**：搬動時不小心改到行為（漏傳一個 state、順序錯）。
- **防護**：①一次一小塊 ②每步 build + 手動點一輪 + `npm run security-check` ③一階段一 commit，可隨時 `git` 退回。
- **正式站**：全程不碰；要上線時整批一起、先問你、且學員無感（純內部整理）。

---

## 六、建議執行順序

1. **先做階段 1 + 2**（純工具 + UI 回饋）——風險極低、立刻見效、先把 page.tsx 減掉約 370 行、建立「抽出去」的節奏與信心。
2. 順了再做**階段 3**（handler 分群，最大槓桿）——一個 hook 一個 hook 慢慢來。
3. 階段 4 之後再評估。

> 每個階段我都會：做完 → build + 本機驗 + security-check → 給你看結果 → 你 OK 才進下一階段。
