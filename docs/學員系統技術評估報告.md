# 技術評估與架構審計報告：NLP 課程計分系統

本報告由 CTO、資深架構師、資安顧問及效能工程師團隊聯合評估，針對本系統在經過近期重構與安全升級後的現狀，進行打分、缺點分析與未來優化建議。

---

## 📊 系統技術維度評分

| 評估維度 | 重構前得分 | 目前得分 | 評級 | 重點變更 |
| :--- | :---: | :---: | :---: | :--- |
| **1. 系統安全性 (Security)** | 2.5 / 10 | **8.5 / 10** | **A-** | 啟用 JWT 真實登入、全 25 表 RLS 安全鎖、防改分 DB Trigger、限流防禦。 |
| **2. 效能與擴充性 (Performance)** | 4.0 / 10 | **8.0 / 10** | **B+** | 建立 18 個關鍵效能索引、學員資料載入分流優化、查詢限制與見證牆分頁。 |
| **3. 代碼維護性 (Maintainability)** | 4.5 / 10 | **8.5 / 10** | **A-** | 巨型 `AdminDashboard` 與 `page.tsx` 瘦身 91%，解耦為 Hooks 與分頁元件。 |
| **4. 測試與交付品質 (QA & CI/CD)** | 1.0 / 10 | **7.0 / 10** | **B-** | 建立 `security-check` (16項) 與 `test-flows` (14項) 自動化自檢腳本。 |

*系統已成功脫離隨時可能被黑客攻擊、或隨人數增加而崩潰的「高風險原型期」，跨入具備生產級安全與效能的「成熟穩定運行期」。*

## 🔍 各維度深度剖析：優點、缺點與隱患

### 一、 系統安全性
*   **👍 優點**：
    1.  **F12 漏洞被鎖死**：過去學員可直接利用控制台修改 `profiles.score` 或偽造 `submissions.status = 'approved'` 自行加分。現已透過資料庫層面的 RLS 政策與 `SECURITY DEFINER` Trigger 徹底防堵。
    2.  **真實身份綁定**：使用真實的 Supabase Session JWT，資料庫能確實校驗操作者的 UID，安全性提升數倍。
*   **⚠️ 缺點與隱患**：
    -   **手機簡訊與密碼缺失**：目前登入仍使用「姓名＋電話」核發 Token。這意味著如果有人知道其他學員的姓名與手機號碼，技術上仍可透過 API 登入其帳號。雖然目前無涉隱私金流，但仍有冒名登入與打卡的風險。
*   **💡 建議**：
    -   **下一階段優化**：考慮整合 LINE Login、微信登入，或是真正的手機 OTP 簡訊驗證碼，實現真正 100% 防冒名的帳號驗證。

---

### 二、 效能與擴充性
*   **👍 優點**：
    1.  **資料分流**：學員登入不再拉取全部 25 張表的歷史資料，改為依據 `batch_id` 進行 scoped 讀取，有效避免頻寬浪費。
    2.  **全表掃描消除**：建立 `idx_submissions_student_id` 等 18 個關鍵索引，確保在高併發打卡時，Supabase 資料庫不會因為全表掃描（Seq Scan）導致 CPU 暴衝。
*   **⚠️ 缺點與隱患**：
    -   **缺乏快取層 (Caching)**：系統中一些不太變動的靜態 lookup 表（如神獸清單 `pets`、進化階段 `pet_stages`、任務範本 `mission_templates`）每次開啟網頁時仍會重複從資料庫拉取，無謂消耗 API 請求數。
*   **💡 建議**：
    -   將靜態常數表在前端進行 localStorage 快取（設定過期時間），或在 Next.js Server Side 進行 API Cache，減少不必要的資料庫往返。

---

### 三、 代碼維護性與可讀性
*   **👍 優點**：
    1.  **極致的瘦身**：成功將高達萬行的兩個巨無霸元件拆解，模組化程度高，新人接手或日後新增功能，能直奔對應的 Tab 元件與 Hook，大幅降低改壞其他地方的機率。
*   **⚠️ 缺點與隱患**：
    -   **狀態傳遞鏈過長 (Prop Drilling)**：因為沒有引入全域狀態管理工具，抽出來的 Hooks（如 `useGameActions`）仍需傳入大量的 `state` 和 `setState` 參數。
    -   **TypeScript 類型警告 (any 殘留)**：部分複雜資料結構在拆分時使用了 `any` 來繞過型別檢查，這在日後重構欄位時可能失去編譯器的保護。
*   **💡 建議**：
    -   **引入 Zustand**：在前端引入輕量級狀態管理工具 **Zustand**，將 `currentUser`、`userEnrollments` 等全域狀態放進 store，徹底消除參數傳遞鏈。
    -   逐步清空程式碼中的 `any` 型別，補齊強型別定義。

---

### 四、 測試與交付品質
*   **👍 優點**：
    1.  **具備防護欄 (QA Guardrails)**：`npm run security-check` 和 `test-flows` 讓每一次改動都有基本的自動化把關，大幅減輕人工驗收負擔。
*   **⚠️ 缺點與隱患**：
    -   **缺乏視覺與 UI 測試**：目前的自動化測試只測試了「資料庫 API 與邏輯流轉」，網頁上的按鈕會不會按下去沒反應、跑版、CSS 抖動，目前自動化測試完全測不到，必須依靠人工肉眼測試。
*   **💡 建議**：
    -   引進 **Playwright** 撰寫 2-3 個關鍵的 E2E 煙霧測試（例如：開啟網頁 ➔ 模擬輸入姓名電話登入 ➔ 檢查是否成功進到首頁 ➔ 模擬點擊打卡），並整合到 GitHub Action 實行自動化 CI。

---

### 五、 資料庫架構健全度
*   **👍 優點**：
    1.  資料庫表結構扁平，PostgREST 對接簡單，擴充欄位非常快速。
*   **⚠️ 缺點與隱患**：
    -   **無資料庫外鍵約束 (No Foreign Key Constraints)**：目前表與表之間完全沒有硬性約束。例如：刪除一個小隊，資料庫不會自動清理該小隊學員的 `team_id`，這會導致產生孤兒資料（Orphaned Data），時間久了資料庫可能會出現髒資料。
*   **💡 建議**：
    -   在資料庫層面補上適度的外鍵約束與聯級操作（例如：`ON DELETE SET NULL` 或 `ON DELETE CASCADE`），以維持資料庫在數年運營後的資料一致性。

---

## 🏁 未來技術路徑演進建議 (Roadmap)

若要繼續提升系統評分至 9.0 (A 級卓越系統)，建議的演進順序為：

1.  **短期 (現下)**：維持目前的 1.8.0 生產環境穩定運行。
2.  **中期**：引入 **Zustand** 重構前端狀態，消除 prop-drilling；同時在資料庫補上**外鍵約束**，防止髒資料產生。
3.  **長期**：引進 **Playwright** 實施 UI 自動化測試；並將登入驗證升級為 **真實手機簡訊驗證 (OTP)**。
