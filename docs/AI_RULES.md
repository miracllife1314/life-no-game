# AI RULES — AI 開發總規則（單一事實來源）

> 本檔是所有 AI 開發規則的**單一事實來源（Single Source of Truth）**。
> `docs/TASK_ROUTER.md` 與 `.claude/skills/*/SKILL.md` 只做指路與摘要，不重抄本檔全文。
> 其他文件引用本檔時，請用「第 N 節」純文字指路。各節標題請保持穩定。

---

## 0. AI Agent 協作與防錯原則 (核心心法)

本專案不依賴 AI 的記憶、自律或臨場判斷來保證正確性。凡是可以用工具、腳本、測試、CI 或權限機制保證的規則，都應優先落實為可執行的硬性護欄。目標是讓錯誤更早、更便宜被修正（落實到自動化 / 測試 / DB 約束），而非要求 AI 永不犯錯。

### 0.1 規則優先機制化
不要只把規則寫在 Markdown 或 Prompt 裡。若某個錯誤會重複發生，應優先改成 ESLint、TypeScript、Prettier、test case、preflight script 等自動化護欄。**Prompt 只能提醒 AI；工具鏈才能真正防錯。**

### 0.2 每一步都必須可驗證
見 §9 第 4 點（Goal-Driven Execution）與 §10（完成的定義）——同一原則統一由該兩節維護，本處不重抄。

### 0.3 禁止危險連續指令
**嚴禁**把格式化、測試、commit、push 串成一行執行（例如 `npm run format && git add . && git commit && git push`）。每一步都要獨立執行，並在關鍵步驟（如 push 前）檢查結果（`git status`, `git diff`）。
同理，**禁止 `git add .` / `git add -A` 整批加入**——逐檔（或逐目錄）確認後加入，避免把無關檔案或機密掃進 commit。（為何無法機器化：hook 只看得到 staging 結果、看不到用了什麼指令，只能靠協定。）

### 0.4 工作範圍先縮小，找不到再擴大
AI 應先從最可能相關的檔案開始檢查，避免一次讀太多無關內容導致注意力稀釋。若證據指向底層（API, RLS, Cache），再主動提出擴大檢查範圍，禁止在缺乏證據時全專案亂改。

### 0.5 高風險任務優先測試驅動
見 §10 第 2 點與 `docs/TESTING.md`（測試策略 SSOT）——「先寫會報錯的測試再寫實作」的原則統一由該檔維護。

## 1. 開發前必做

1. 先讀 `docs/TASK_ROUTER.md`。
2. 判斷本次任務類型。
3. 根據任務類型讀對應文件。
4. 不得盲目全讀。
5. 不得完全不讀規則直接修改。
6. 小改走 Tiny Change Fast Path（第 2 節）。
7. Prototype 只允許做草稿與概念驗證，不得混入正式功能（第 3 節）。
8. 中改在準備動手改程式前使用 Technical Review Mode（第 4 節）。
9. 高風險先停手，提出方案與風險（第 5 節）。
10. 每次完成任務都要檢查是否需要同步更新 docs（第 8 節）。

---

## 2. Tiny Change Fast Path

小改不需要冗長分析，不需要角色審查。

只需回報：

1. 修改內容。
2. 修改檔案。
3. 測試方式。
4. 風險提醒。

---

## 3. Prototype Mode

允許在早期討論或畫面草稿階段使用 Prototype Mode。

限制：

1. 只允許做 UI 草稿、流程討論、靜態 mock、概念驗證。
2. 不得連線資料庫。
3. 不得修改 migration。
4. 不得修改 RLS。
5. 不得修改權限。
6. 不得修改 production。
7. 不得建立真實付款、點數、優惠碼、webhook 邏輯。
8. 不得把 prototype 假資料混入正式功能。
9. 一旦 Prototype Mode 要轉成正式功能，必須重新走 `docs/TASK_ROUTER.md` 任務分類與正式規範。

---

## 4. Technical Review Mode

中改若只是討論想法，不需要強制格式。

但在準備動手修改程式前，必須從以下維度簡短評估：

1. **Security**：是否有 RLS、權限、個資、service_role key 風險？
2. **Database**：是否需要 migration、index、constraint、資料結構調整？
3. **Performance**：是否有 N+1、一次撈全表、缺分頁問題？
4. **UX**：是否影響手機版、表單、錯誤提示、操作流暢度？
5. **QA**：需要測哪些正常流程與錯誤流程？
6. **Rollback**：如果出錯，如何回復？

> 本節就是**角色檢查的日常執行體**：六個維度即六個角色視角
> （Security→資安、Database→資料庫、Performance→開發、UX→UI/UX、QA→測試、Rollback→DevOps，對照 `docs/ROLE_SYSTEM.md` §3）。
> 日常不開多角色會議——中改做完這六維即可動手；
> 高風險與集中審查的獨立審查載體見 §12 與 `docs/ROLE_SYSTEM.md` §7。

---

## 5. 高風險熔斷

**判斷原則：改「規則與結構」= 高風險；在既有規則內「使用」資料 = 中改。拿不準時一律升級為高風險。**

### 高風險（必須停手，先提出方案與風險，不得直接修改）

1. 資料庫 **schema 變更**：新增 / 修改 / 刪除資料表、欄位、enum、constraint、index、trigger、function、RPC（即任何 migration）。
2. RLS 新增或修改。
3. auth / login 流程變更。
4. admin / role / permission 變更。
5. production / deployment。
6. payment / refund / coupon / points / ledger / webhook 等**金流與交易的寫入邏輯**。
7. 個資**寫入、更新、刪除**、個資可見範圍變更。
8. file upload / storage 的權限與規則。
9. 大量資料更新 / 大量資料刪除。
10. leaderboard 等大規模批量計算的**首次架構設計**。

### 中改（做完 Technical Review 即可執行，不需停手）

1. 使用**既有**資料表與**既有** RLS 做一般讀取 / 顯示功能。
2. 在既有權限邊界內的一般 CRUD 功能開發——前提：不改 schema、不改 RLS、不碰金流寫入、不碰個資寫入更新刪除。

> 為什麼要分：商用系統多數功能都會「碰到」資料庫。若一律熔斷，流程會被疲勞繞過，規則等於失效。熔斷保留給真正改變系統規則與結構的操作。

高風險任務必須先提出：

1. 方案 A。
2. 方案 B。
3. 推薦方案。
4. 風險。
5. 回滾方式。
6. 具體測試方式。
7. 需要使用者確認的事項。

高風險改動（migration / auth / 金流 / 權限）實作完成後、交付前，
執行 `bash scripts/review.sh` 交乾淨獨立 AI 實例審查（`docs/ROLE_SYSTEM.md` §7）。
目前為**軟跑階段**：結果供人決策，不掛 pre-push 硬門。

## 6. 執行中自動升級

如果一開始判斷為小改、Prototype 或中改，但執行過程中發現需要碰到高風險內容，必須**立刻停止**，升級為高風險任務（第 5 節）。

不得因為已經開始修改，就繼續硬做。

---

## 7. 禁止繞路

AI 嚴禁為了避免 migration 或高風險審查，而用以下方式繞路：

1. 把結構化資料硬塞進 metadata。
2. 濫用 jsonb 欄位。
3. 用 localStorage 假裝保存正式資料。
4. 硬塞到不相關欄位。
5. 用前端狀態假裝資料已保存。
6. 用註解或 TODO 取代正式架構。
7. 為了避開 RLS 而改用 service_role。
8. 為了避開資料表設計而塞字串。
9. 為了避開權限設計而只做前端隱藏。
10. 為了避開正式流程而混入假資料。

若正確做法需要 migration，必須誠實回報，不得繞路。

## 8. 文件同步熔斷

AI 每次修改程式、資料庫、UI、流程、權限、部署或任何系統行為時，必須檢查該修改是否會讓 `docs/` 下的文件過期。

若會影響文件，必須在**同一次任務**中同步更新相關文件，不得留到以後。

常見對應：

1. 修改資料表、欄位、RLS、RPC、trigger、index
   → 檢查並更新 `docs/DB_DESIGN.md`、`docs/DB_MIGRATION_RULES.md`、`docs/SECURITY_CHECKLIST.md`

2. 修改功能流程、前台、後台、角色操作
   → 檢查並更新 `docs/FEATURE_MAP.md`、`docs/PROJECT_CONTEXT.md`

3. 修改付款、點數、優惠碼、webhook、報名、退款
   → 檢查並更新 `docs/TRANSACTION_RULES.md`、`docs/INCIDENT_PREVENTION.md`

4. 修改 UI、品牌、表單、後台編輯器、前台轉化流程
   → 檢查並更新 `docs/BRAND_UI_SYSTEM.md`、`docs/UX_GUIDELINES.md`

5. 修改部署、env、production、staging、rollback
   → 檢查並更新 `docs/DEPLOYMENT.md`、`docs/OBSERVABILITY.md`

6. 修改任何 `.claude/skills/*/SKILL.md`
   → **必須同步鏡像到 `.agents/skills/` 同名檔**（`.claude/` 是權威、`.agents/` 是供其他 AI 工具讀的鏡像；兩者分岔即為第 13 號事故「雙頭馬車」）。

7. 每次完成任務都必須更新 `logs/ai_sync_log.md`。

如果 AI 判斷不需要更新文件，必須在交付回報中明確寫：

```text
本次修改不影響既有 docs 文件，原因是：___
```

### 單一事實來源：不得平行維護兩套規範

1. 每個主題的規則**只能有一份權威文件**。`docs/AI_RULES.md` 是開發流程規範的唯一事實來源。
2. **嚴禁「文件雙頭馬車」**：不得讓兩份文件同時定義同一主題的規則（例如兩份 AI_RULES、兩套 DB 規範）。時間一久兩份必然分岔，AI 讀到過期那份就會依「不存在的現況」誤判。
3. brownfield 匯入或既有專案已有規範時：**指定一份為權威**,其餘在檔頭標註「歷史參考,現行規範見 `docs/AI_RULES.md`」,並**停止平行維護**——不再兩邊都改。
4. 發現同主題有多份規範文件時，AI 必須主動提報，建議整併或標註取代，不得默默沿用其中一份。

## 9. Karpathy 守則

必須遵守：

1. **Think Before Coding**：不要亂猜，不確定要標記。
2. **Simplicity First**：用最小可行解法。
3. **Surgical Changes**：只改必要範圍。
4. **Goal-Driven Execution**：每次都要有可驗證成功標準；不能只說「已完成」，要說明做了什麼、怎麼測、有什麼風險。
5. **Keep Folders Lightweight（防目錄與檔案巨大化）**：禁止將所有程式與邏輯堆積在單一目錄下。若單一資料夾下檔案過多（如超過 10 個），或單一檔案行數過長（如超過 300 行），必須主動進行模組化拆分（拆成子目錄、獨立 Hooks、子組件或專屬資料存取層），維持結構輕量與清晰。

## 10. 完成的定義（Definition of Done）

任何程式修改，回報「**完成**」之前必須全部確認：

1. **build / typecheck 通過**（專案已可建置時）。**「build」= 專案完整 build 指令（如 `npm run build`），不是只有 `tsc --noEmit`**——Vercel 把 prettier/eslint 排版錯誤當 build 失敗且**靜默不部署**（繼續服務舊版），tsc 和 dev server 都抓不到。本機檢查採「快慢分層」：輕量快檢在 pre-commit、完整 build 在 pre-push（規則詳見 `docs/DEPLOYMENT.md` 快慢分層節）。
2. **相關測試通過**；金流、權限、交易類的新邏輯**必須**有對應測試（策略見 `docs/TESTING.md`）。
3. **`bash scripts/preflight-check.sh` 通過**，無紅線。
4. **實際執行過受影響的流程**——真的跑過畫面或 API，不是只看程式碼覺得對。
5. **文件同步熔斷檢查完成**（第 8 節），`logs/ai_sync_log.md` 已更新。

### 回報用語協定（三個詞不得混用）

| 用語 | 條件 |
|---|---|
| **「完成」** | 上列 1~5 全過，**含實際跑過端到端流程** |
| **「已部署，待驗證」** | build/push 過了，但真實流程還沒跑（或還有使用者要做的前置步驟） |
| **「已修改，待驗證」** | 改了但連部署驗證都還沒有 |

另外：**還在使用者手上的前置步驟（如：待套 SQL、待實測付款），必須列在回報的最上方**，不得藏在文末。

**AI 無法自行執行的流程，必須明講「我測不了 X，需要你測 Y」，不得含糊帶過。**
系統應提供 AI 可用的端到端驗證通道（必備基建，見 `docs/TESTING.md` 第 0 節）——
沒有通道，第 4 點「實際執行過流程」物理上做不到，永遠只能報「已部署，待驗證」。

### 🛡️ 載入標示協定（可抽查的驗證點）

**實際載入**某守衛（`.claude/skills/*/SKILL.md`）或角色檔（`docs/roles/*.md`）時，
在該次回報開頭標一行：`🛡️ 已載入：<skill 名或角色檔名>`（多個以逗號分隔）。

1. **只在真的讀了檔案時才標**——這一行是使用者的抽查點（可追問「你載入的清單第 N 條是什麼」驗證），沒載入就標視同謊報完成。
2. 沒有載入任何守衛／角色檔的一般任務，**不需要**任何開頭宣告。
3. 舊制「每次回報開頭全域宣告防護狀態」已於 v1.7.0 廢除：實戰一週 0% 執行率，
   留著只會損傷整套規則的權威。
   （為何無法機器化：AI 是否真的讀了檔案，script 攔不到，只能縮小成可抽查的語言協定。）

> 這一節的目的：把「AI 覺得寫完了」和「系統真的能動」分開。前者是程式碼，後者才是工程。
> （用語協定來源：實戰回饋——「build 綠 + 已 push」被系統性當成「完成」回報，使用者其實還要套 SQL 和實測。）

## 11. 自主執行與效率原則

規範的目的在保護品質，不在製造來回。**在規則邊界內，AI 應自主執行到底：**

1. **自主執行**：規則內能自己判斷的事直接做完，不逐步請示。只有兩種情況才停下來問：
   (a) 觸發第 5 節高風險熔斷；(b) 需要使用者做安全或商業決策。
2. **批次回報**：一次任務一份總結，不每做一小步就回報一次。長任務先列計畫，然後一口氣執行完再統一交付。
   **具體動作標準：一個功能 = 一個 commit = 一次 push**。禁止同一功能連續推多個 commit——會造成部署空窗，使用者點開就撞到 404 / 裸樣式。（來源：實戰回饋，此問題重複發生。）
   > 補註：本條防的是「**同一批已知改動**拆多次推」造成的部署空窗；**不禁止**「部署 → 使用者實測 → 修正」的迭代循環——那是健康的開發節奏。每輪修正本身仍應累積成一次推。
3. **省 token**：依 Router 只讀需要的文件；同一對話中已讀過的內容不重讀；輸出精準，不重複貼大段已知內容。
4. **批次回饋**：模板回饋（`logs/template_feedback.md`）累積成批再回流母版，不逐條往返。
5. **效率永遠讓位於安全**：省的是「來回與廢話」，不是「preflight、測試、高風險熔斷」。任何檢查步驟不得為省時間而跳過。

> 這一節的來源：真實使用回饋——「來來回回很花時間跟額度」。規則成熟後就該放手自主，事事確認反而是另一種浪費。

## 12. 多角色協作

角色系統的單一事實來源是 `docs/ROLE_SYSTEM.md`，完整角色說明在 `docs/roles/`（按需載入）。本節只放鐵律：

1. **依任務與風險動態啟動角色，不得每次所有角色全部啟動**（小改單角色直做；中改功能段落完成後集中審查一次；高風險依風險啟動相關專業角色）。
2. UI/UX 商用產品設計師的「必須啟動」清單見 `docs/ROLE_SYSTEM.md` §4；新頁面 / 新流程未指定風格時，先依 `docs/STYLE_PACKS.md` 提問，不得自行猜測風格。
3. 角色分工不得混為一談（例：UI/UX 負責體驗與視覺驗收，前端 Developer 負責程式實作）。
4. 集中審查採**獨立審查制**，執行載體是**乾淨的獨立 AI 實例／子代理**（只給 diff＋需求，不給實作脈絡；「同一 AI 分飾多角開會」已退役）：實作者不得批准自己的成果；資安 / 資料庫 / 金流 / 核心需求 / 嚴重手機 UX 的阻擋不受多數決蓋過；無證據標 `Unverified`（完整機制見 `docs/ROLE_SYSTEM.md` §7；機器化入口 `bash scripts/review.sh`，目前軟跑階段）。
