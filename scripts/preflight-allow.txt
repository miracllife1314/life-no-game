# preflight-allow.txt — RLS 例外白名單（給 migration 檔用）
#
# ⚠️ 重新同步母版時：本檔「只更新上方說明、保留你的例外行」，
#    絕不可用母版空白版直接覆蓋——那會洗掉你已審核的例外，preflight 立刻變紅。
#    （本檔屬「專案內容」，非「純模板」；同 PROJECT_CONTEXT 等回填檔一樣，合併不覆蓋。）
#
# 用途：登記「已人工審核、刻意放行」的 RLS 政策紅線（如公開/參考資料表的
#       USING (true)）。preflight-check.sh 會放行這裡列出的行。
#
# 為什麼需要這個檔（而不是在 migration 裡加 -- preflight-allow 註記）：
#   migration 檔「只增不改」，在裡面加註記會破壞該鐵律、還會觸發 preflight
#   檢查 #6「既有 migration 被修改」。所以 migration 的例外抽到這個外部檔。
#   （非 migration 檔仍可用 inline 的 `-- preflight-allow: 原因` 註記。）
#
# 格式：每行一筆「路徑:行號   原因」
#   - 路徑要跟 preflight 輸出的一模一樣（相對專案根，例 supabase/migrations/xxx.sql）。
#   - 行號是該政策在檔案中的行號（migration 不改 → 行號永久穩定）。
#   - # 開頭為註解；空行忽略。
#
# 鐵律：只放「確認永遠不含機密/個資」的公開或參考資料表。
#       機密（sk_live 等）不受本白名單影響，列了也照攔。
#       每筆都應寫清楚原因與審核日期，供 git 稽核。
#
# 範例（實際使用時取消註解並改成真實路徑行號）：
# supabase/migrations/0001_init.sql:26   member_tiers 參考表，僅唯讀公開資料，已審核 2026-07-05
# supabase/migrations/0003_badges.sql:12 badges 參考表，僅唯讀公開資料，已審核 2026-07-05
# =============================================================
